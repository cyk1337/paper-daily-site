Tokenizer Scaling：从 32K 到 128K 之外——被当常量的那根杠杆

Tokenizer Scaling：从 32K 到 128K 之外——被当常量的那根杠杆·14 篇论文·4 个子主题·2026年4月19日

作者@Thor·gpt-5.2

领域综述

Tokenizer 过去常被当作“预处理细节”：只要 coverage 够、bytes/token 低就行。但近三年的受控证据把它推回到一等规格：固定 compute 下，仅 tokenizer 选择就能带来 0.6–5.1 pp 的下游差异 [2]；把 vocab 从 32K 扩到 128K，在大模型推理侧几乎不增加算力，却能把训练 loss 拉低 0.02–0.04 nats，并显著改善 non-English 与 code 的 bytes/token [1]。更关键的是，tokenizer 不只是压缩器：数字 tokenization 会在 3–5 位算术上造成 10–20 pp 的稳定差距且不随 scale 收敛 [7][5]；而 under-trained tokens 在已部署模型中普遍存在，会在生成时“吐垃圾”，需要在训练收尾做系统性扫描与修复 [3]。跨 tokenizer 的比较若仍用 per-token PPL，会把“token 变短”误当成“模型变好”；更稳的做法是统一用 bits-per-byte（BPB）做归一化 [9]。tokenizer-free（byte/char/patch）路线在 BPB 上已接近或达到 BPE parity [10][11][6]，但 2–3× 推理成本仍让它更像研究方向而非 2026 年的默认生产方案。结论层面的建议：把 tokenizer 当成 pretrain 产品规格的一部分，像数据混合与训练配方一样被版本化、被诊断、被回归测试。

把 tokenizer 从“默认继承”升级为可回归的 pretrain 产品规格：用 BPB 说话，用 Magikarp 修债

一句话导读

结论：tokenizer 不是预处理常量，而是固定 compute 下能稳定撬动 0.6–5.1 pp 下游差异的一等变量 [2]。vocab 从 32K 扩到 128K 在推理侧几乎不加成本，却能带来 0.02–0.04 nats 的 loss 下降，并改善 non-English 与 code 的 bytes/token [1]。数字 tokenization 会在 3–5 位算术上造成 10–20 pp 的长期差距，且 scale 不会自动抹平 [7][5]。训练好的模型里普遍存在上千个 under-trained tokens，会在部署期生成垃圾，需要在收尾用 Magikarp 扫描并做短 continued pretrain 修复 [3]。跨 tokenizer 的比较必须用 bits-per-byte（BPB）归一化，per-token PPL 会把“token 变短”误当成“模型更好” [9]。tokenizer-free（ByT5/MEGABYTE/BLT）在 BPB 上已接近 parity，但 2–3× 推理税让它更适合作为研究与特定场景方案，而非 2026 年默认生产路线 [10][11][6]。

核心断言

#1在固定 compute 的受控预训练里，仅 tokenizer 选择就能造成 0.6–5.1 pp 的下游差异，量级与常见 data mixture 改动相当 [2]。
#2把 vocab 从 32K 扩到 128K 可带来约 0.02–0.04 nats 的训练 loss 下降，同时在大模型推理侧几乎不引入额外算力瓶颈；收益主要体现在 non-English 与 code 的 bytes/token 改善 [1]。
#3数字 tokenization（single-digit vs merged BPE）会在 3–5 位算术上造成 10–20 pp 的稳定差距，并且从 1B→7B 到 frontier 模型都不显示“随 scale 收敛”的迹象 [7][5]。
#4已训练模型中可检测到 3k–10k+ 的 under-trained tokens，这些 token 在生成时更容易输出乱码/无意义片段；用 Magikarp 的 embedding-gap + probe 可在部署前定位风险点 [3]。
#5跨 tokenizer 的可比指标应以 byte 归一化（BPB/byte-normalized loss）为主；per-token PPL 会系统性偏向更高压缩率的 tokenizer，即使其下游并不更好 [9][8]。

§1 共同底层事实：tokenizer 改变的是“学习信号的几何形状”，不只是压缩率

先给一个可操作的共识：tokenizer 的主要作用不是把文本压得更短，而是决定“哪些模式被当作原子、哪些需要模型去组合”。这解释了为什么仅看 fertility/bytes-per-token 往往会误判。Schmidt et al. [8] 在固定 compute 下展示：更高压缩率的 tokenizer 并不稳定带来更好下游，因为 morphology 对齐、domain coverage、以及 token 频次分布会改变梯度信号的稀疏性与可泛化的组合结构。Hiraoka et al. [19] 进一步从表示层面补了一块：embedding 会编码 token 的表面形态（长度/字符形状等），意味着 tokenizer 直接改变了模型的表征基元。

Ali et al. [2] 把这些直觉落到受控实验：固定模型规模与训练预算，换 tokenizer 就能产生 0.6–5.1 pp 的下游差异。这个量级之所以“便宜”，原因在于 tokenizer 改动几乎不触碰训练系统的主成本（算子、并行、KV cache），却能系统性改变序列长度分布、稀有 token 的学习难度、以及跨域/跨语种的覆盖方式。把 tokenizer 当常量的代价，是把一部分可控的误差留到后面用更贵的 compute 或更复杂的后训练去补。

tokenizer 决定哪些模式是“原子”，从而决定哪些能力是组合泛化、哪些变成记忆负担。

§2 32K→128K：更像“把容量从序列长度挪到 embedding 表”，而不是单纯压缩

关于 vocab 扩容，一个更稳的读法是：它把一部分建模负担从“更长的序列组合”转移到“更大的 embedding 表 + 更直接的子串覆盖”。Llama 3 报告把 vocab 从 32K 扩到 128K 后，训练 loss 下降约 0.02–0.04 nats，并给出 fertility 下降等统计 [1]。DeepSeek-V3 把 tokenizer 设计写进技术报告，并用 BPB/相关表格把收益落到可比较的指标上 [4]。这两份报告共同指向一个工程事实：当模型规模与服务栈已经为大 batch/大 KV cache 优化时，128K vocab 的额外成本主要是 embedding/softmax 的参数与带宽，而不是推理算力的主瓶颈；相对地，序列变短会直接减少 attention 相关成本。

但“vocab 越大越好”并不成立为无条件规律。Schmidt et al. [8] 提醒：压缩率不是充分指标；更大的 vocab 也更容易制造长尾 token，导致训练信号稀薄。这个风险在 Magikarp 的诊断里会被放大：under-trained tokens 往往集中在低频、形态怪异或跨域拼接的 token 上 [3]。因此，32K→128K 的可复制收益更像是“在合理的语料抽样与过滤下，把常见子串覆盖做得更细”，而不是盲目把 vocab 推到 256K+。目前公开文献对 128K→256K 的 fixed-compute/latency 证据仍缺口（见 open questions）。

§3 能力缺口的来源：数字、多语言、代码都在惩罚“边界不对”的 tokenizer

数字是最干净的反例：它几乎不依赖语义，只考验组合与位置泛化。Spiess et al. [7] 在受控设置下显示 single-digit tokenization 在 3–5 位算术上领先 merged BPE 10–20 pp，且从 1B 到 7B 不收敛；Singh et al. [5] 把同一排序扩展到 GPT-4、Claude、Gemini 等系统，说明这不是“小模型训练不够”的问题，而是 token 边界把“多位数”变成了更难学习的离散词表记忆。Nogueira et al. [17] 也从更早的算术任务角度支持：数字的表面形式会改变学习难度。

多语言场景里，错误边界的表现更隐蔽：不是某个任务掉点，而是某些语言被系统性分配到更差的表示基元。Limisiewicz et al. [13] 指出 vocab 槽位分配与跨语言 overlap 会影响低资源语言的有效容量；Toraman et al. [14] 在土耳其语这类 morphology 丰富语言上展示 tokenization 选择的敏感性。代码则把“空白符边界”推到前台：缩进与对齐是语法的一部分，若 tokenizer 把空格与相邻 token 合并，会让模型在学习缩进模式时付出额外的组合成本；这类动机在代码模型与数据集工作中反复出现 [16][15]。把这些现象放在一起看，会得到一个统一机制：边界错误会把本该可组合的结构（digit、morpheme、indent）变成稀疏的长尾 token，从而制造“scale 也不一定能抹平”的能力缺口。

§4 指标与路线选择：BPB 解决“可比性”，推理税决定 tokenizer-free 的落点

跨 tokenizer 讨论最常见的技术性误会，是用 per-token PPL/CE 做比较：token 变短会机械性降低 per-token loss，但这不等价于对原始文本（byte 序列）的更好建模。更稳的做法是把损失按 byte 归一化，即 BPB/byte-normalized loss；这类 byte 口径的数据计量与比较思路在 The Pile 的工程化叙述中已经出现 [9]，并在后续 tokenizer 研究中被反复引用为跨 tokenizer 的共同坐标系 [2]。

在此坐标系下，tokenizer-free 的进展可以更清晰地被定位：ByT5 展示 byte-to-byte 预训练能在质量上接近 subword，同时带来噪声鲁棒性与多语覆盖的结构性优势 [10]；MEGABYTE 用 multiscale 架构把 byte-level 推到更长序列 [11]；BLT 则用基于局部预测熵的动态 patching 把 BPB 拉到 BPE parity，并明确报告推理侧 2–3× 成本 [6]。因此，2026 年更务实的路线选择不是“要不要 tokenizer-free”，而是“在哪些产品约束下值得交 2–3× 吞吐税”：例如极端多语/噪声输入、对 OOV 极敏感的场景，或需要统一处理文本+二进制片段的输入管线。对通用对话/代码助手这类吞吐敏感产品，先把 BPE 做对（vocab 64K–128K + 诊断修债 + BPB 指标）通常更划算 [1][3][4]。

研究立场对比

阵营 A：tokenizer 是冻结的预处理，只要 coverage 够就行

立场 — 默认沿用主流 BPE/WordPiece 配置，把优化预算放在 data mixture、训练配方与后训练；tokenizer 只做最小化的脚本覆盖与特殊符号处理。

证据：[21][22][23]

反方 — 受控实验已经显示 tokenizer 不是“二阶细节”：固定 compute 下 0.6–5.1 pp 的差异无法用“coverage 已经够了”解释 [2]；而且 under-trained tokens 在多款模型中普遍存在，属于“默认沿用”会积累的训练债务 [3]。

判词 — 更务实的建议是把 tokenizer 升级为可回归规格：至少在每次大规模 pretrain 前做一次 tokenizer sweep（小模型/短跑），并在训练收尾做 Magikarp 扫描 [2][3]。

阵营 B：vocab 越大越好，直接冲到 256K+

立场 — 把更大 vocab 视为近似“免费压缩”：序列更短、attention 更省，收益会单调增加；embedding/softmax 的额外参数相对可忽略。

证据：[1][4]

反方 — 现有公开证据主要覆盖到 128K，而不是 256K+；同时更大 vocab 会放大长尾 token 与 under-trained 风险 [3]，且压缩率本身并不能预测下游 [8]。缺少的是 32K/64K/128K/256K 在 fixed compute 下的系统对比（含 latency、embedding 带宽、under-trained token 率）。

判词 — 更稳的落点是 64K–128K 作为默认区间，并把“更大 vocab”当作需要证据支撑的产品实验，而不是默认信条 [1][8]。

阵营 C：tokenizer-free 才是终局，应该放弃 BPE

立场 — 用 byte/char/patch 直接建模，消除分词偏置与 OOV，获得更好的鲁棒性与多语公平性；把系统复杂度交给模型结构解决。

证据：[10][11][6][12]

反方 — BLT 已把质量推到 BPE parity，但同时明确给出 2–3× 推理成本 [6]；对吞吐敏感的产品，这个税往往比 tokenizer 带来的 0.6–5.1 pp 机会成本更“硬” [2]。此外，很多生产问题（数字、空白符、长尾 token）在 BPE 路线下可以用更小的工程改动修到可接受水平 [7][3]。

判词 — 更务实的定位：tokenizer-free 作为“鲁棒性/多语公平性优先”的分支路线持续投入；通用生产默认先把 BPE 做对，并用 BPB 与推理吞吐共同做门槛 [6][9]。

阵营 D：tokenizer 是 pretrain 产品规格——先把 BPE 做对，再谈替代

立场 — 把 tokenizer 版本化并纳入回归：vocab 64K–128K、数字 single-digit、空格/换行独立 token；训练后期用 Magikarp 扫 under-trained tokens 并做短 continued pretrain；跨 tokenizer 一律报 BPB。

证据：[2][1][4][3][9][7]

反方 — 主要挑战不是理论，而是组织与工具链：tokenizer sweep、BPB 报告、Magikarp 扫描与修复需要进入训练流水线；同时多语言 vocab 分配与代码空白符策略需要与数据混合共同设计 [13][16]。

判词 — 结论层面的建议：把阵营 D 当作默认工程路线；tokenizer-free 作为并行研发分支，用“BPB parity + ≤1.3× 推理成本”作为进入通用生产的硬门槛 [6][9]。

实践要点

一条可执行的清单（带边界与引用）：
1) DO：vocab 默认选 64K–128K；把 256K+ 当成需要单独 fixed-compute + latency 评估的实验，而不是默认升级 [1][8]。
2) DO：数字用 single-digit（0–9 独立 token），并避免把多位数常见片段 merge 成长 token；算术差距在 1B→7B 与 frontier 都不显示收敛 [7][5]。
3) DO：空格与换行保持独立 token，不与相邻词合并；代码场景额外加入 multi-space token（至少覆盖 2/4/8 spaces），减少缩进模式的组合负担 [4][16]。
4) DO：tokenizer 训练语料按“最终 pretrain mixture”抽样，而不是按 raw crawl 比例；规模上 200–500GB 往往足够覆盖高频子串，剩余依赖训练期学习 [2]。
5) DO：训练 tokenizer 前先做 document-level MinHash + line-level exact dedup，并使用不弱于 pretrain 的质量过滤；否则 vocab 槽位会被模板噪声与重复片段占用，放大长尾与 under-trained 风险 [9][3]。
6) DO：多语言 tokenizer 做 vocab 分配时显式监控“语言槽位占比/overlap”，避免英语或高资源语言吞掉过多槽位；把分配指标纳入回归 [13][14]。
7) DO：每次大规模 pretrain 收尾跑 Magikarp，列出 under-trained tokens 清单；对出现次数 <100 的 token 做短 continued pretrain（经验上先从 ≤5B tokens 的补训预算起步），并把“修复前后 probe 通过率”纳入发布门槛 [3]。
8) DON’T：跨 tokenizer 用 per-token PPL 做结论；必须报告 BPB/byte-normalized loss，并在同一 byte 口径下对齐训练步数与 compute [9][2]。
9) OPEN（证据不足）：是否需要把 tokenizer 与模型联合优化（而非冻结）取决于组织成本与收益曲线；现有方法学存在，但缺少在 trillion-token 级别的公开对照 [18]。

悬而未决的问题

  • Q1.哪些工作在 fixed compute 下直接比较 32K/64K/128K/256K vocab，并同时报告推理 latency、embedding/softmax 带宽成本、以及 under-trained token 率？
  • Q2.有哪些明确证据表明 under-trained/dead tokens 在已部署模型中普遍存在，并且可以通过“针对稀有 token 的 continued pretrain”稳定修复（含修复预算与回归指标）？
  • Q3.哪些 tokenizer-free 工作在报告 BPB parity 的同时，给出可复现的推理成本/吞吐惩罚（例如 tokens/s、FLOPs、KV cache 压力）？
  • Q4.代码域是否存在在 matched compute 下直接比较 BPE、unigram、byte、character tokenization 的研究（含空白符策略），并报告代码生成/理解的差异？
  • Q5.有哪些论文或技术报告明确主张用 BPB/byte-normalized 指标替代 per-token PPL 来做跨 tokenizer 评估，并通过实证展示两者排序不一致的案例？
