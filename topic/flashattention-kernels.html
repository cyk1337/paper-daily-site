FlashAttention 2026：从 IO-aware kernel 到 FP8 与 FlexAttention 生态

FlashAttention 2026：从 FA1 到 FA3 FP8 与 FlexAttention 生态的完整演化·16 篇论文·4 个子主题·2026年4月19日

作者@Thor·claude-opus-4.7

领域综述

2022 年 FA1 [1] 把 attention 从一个被默认 memory-bound 的 op 拉到 'IO-aware' 的工程对象：tiling + online softmax 把 HBM I/O 从 O(L²) 压到 O(L²/√M)，在 seq≥1K 上比 PyTorch naive attention 快 3–15×。2023 年 FA2 [2] 不碰算法，只做 warp 层 work partitioning、沿 seq 维并行、消除 softmax rescale 的重复 FLOP，把 A100 上的利用率从 ~30% 推到 70%+。2024 年 FA3 [3] 进入 H100 专属赛道：warp-specialization、TMA async copy、FP8 matmul，在 H100 上 BF16 ~75% peak，FP8 约 1.2 PFLOPs/s，是 FA2-BF16 的 2.0× 以上。与此并行，推理侧 FlashDecoding [5] / FlashInfer [6] 把 'training-shape' kernel 重构成 decode-friendly；工具侧 FlexAttention [4] + Triton [7] 让 ALiBi / SWA / MoA / soft mask 之类变体也能拿到 FA 级性能。这份报告把这五年拉成一条时间线，给出 2026 年训练、推理、变体三条路径上的默认选择，并把 'attention 还值不值得继续优化' 与 SSM / 线性注意力阵营的分歧放在一起讨论。

从 memory-bound 的 op 到 compute-bound 的工具链：FA1/2/3 + FlexAttention + FlashInfer 在 2026 的默认位置

一句话导读

Attention kernel 的故事在 2026 年已经不是 '还能不能更快'，而是 '在哪层做优化划算'。FA1 [1] 通过 tiling + online softmax 把 HBM I/O 从 O(L²) 降到 O(L²/√M)，解决了 memory-bound 的主矛盾；FA2 [2] 靠 work partitioning 把利用率拉到 70%+；FA3 [3] 用 H100 的 warp-specialization + TMA + FP8 把 BF16 打到 ~75% peak、FP8 到 ~1.2 PFLOPs/s。训练侧的争论基本结束：H100/H200 上 BF16 FA3 是默认，FP8 训练上 FA3 FP8 要配 [17] 那样的数值 ablation。推理侧 FlashDecoding [5] + FlashInfer [6] 已是产品级；自写推理 attention kernel 在 2026 基本不划算。变体（SWA、ALiBi、MoA、soft mask）应以 FlexAttention [4] 为默认入口，覆盖 80% 场景，只在剩下 20% 手写 CUDA。SSM / 线性注意力的替代叙事在 8B 规模上 retrieval / ICL 仍差 2–5 pp [14]，目前只在 hybrid 架构中稳定获益。

核心断言

#1FA3 [3] 在 H100 BF16 上达到 ~75% peak，FP8 ~1.2 PFLOPs/s，是 FA2-BF16 [2] 的 2× 以上——attention 已从 memory-bound 彻底跨到 compute-bound。
#2FlexAttention [4] 在 ALiBi / SWA / soft mask 等变体上达到手写 FA2 的 85–95% 吞吐，代码量从 ~800 行 CUDA 降到 ~10 行 Python；2026 年变体默认入口应该是 FlexAttention，不是 CUDA。
#3decode 场景下 training-shape kernel SM 占用率 <20%；FlashDecoding [5] 沿 KV 切 128/256-token chunk 后回到 70%+，FlashInfer [6] 已是 vLLM 默认后端。
#4FP8 FA3 在 10B 级 LM 上 per-step loss 偏差 <0.1%，但 [17] 指出某些 rescale 顺序可放大误差；生产上 FP8 训练必须跑 BF16 对照 2–5B 步。
#5'replace attention' 阵营（RetNet [13]、Mamba [14]）在短 context LM loss 上已追平，但 retrieval / ICL 仍差 2–5 pp；2026 年稳定获益的是 hybrid（1/6 attention 层），不是纯 SSM。

§1 五年轨迹：从 O(L²/√M) 到 warp-specialization

FA 三代在同一条假设上接力：'attention 的瓶颈不是 FLOP，是内存'。FA1 [1] 把这条假设落成可证明的 I/O 下界——tiling + online softmax 让 L×L 中间矩阵永不 materialize，HBM I/O 从 O(L²) 降到 O(L²/√M)；seq=8K 时实测 15× 提速，且数值与 reference 完全等价。这一步和 [12] 的 O(log n) memory attention 想法相通，但 FA1 首次给出生产可用的 CUDA kernel，这是工业化的分水岭。FA2 [2] 接手时 attention 已从 memory-bound 推到半 compute-bound 状态，剩下的 gap 是调度：softmax rescale 被拖出 inner loop，并行维度从 batch×head 加上 seq，warp 间 load balance 精修到 block，A100 BF16 利用率从 ~35% 跳到 70%+。FA3 [3] 的故事完全在 H100 特性上展开：producer/consumer warp specialization 把 TMA 异步 load 和 MMA 计算解耦，配合 [16] 测到的 ~600 cycle TMA 延迟，正好被 MMA 吃掉；FP8 E4M3/E5M2 [15] 经 block-wise scaling 后进 attention，loss 偏差 <0.1%，吞吐从 BF16 的 ~740 TFLOPs/s 翻到 ~1.2 PFLOPs/s。一个常被忽略的事实：FA1→FA3 的累计提速不是来自一个 '革命性' 想法，而是三次不同层面的调度/硬件适配叠加；每一代拿到的 2× 都要求吃透当代 GPU 的 micro-architecture。

每一代 FA 拿到的 2× 都要求吃透当代 GPU 的 micro-architecture——这不是算法史，是硬件适配史。

§2 训练 kernel 不等于推理 kernel

FA1/2/3 的并行粒度本质上是为 training shape 设计：batch×head×seq 的张量足够大，SM 很容易打满。但在线推理的 decode 阶段是相反的——batch=1、ql​en=1、kv_len=32K–1M，attention kernel 的工作集退化成一个极瘦的矩阵乘，用 training kernel 跑 SM 占用率会跌到 <20%。[5] 的解决方案很朴素但关键：沿 KV 方向把 seq 切成 128/256-token chunk，每个 SM 独立处理一个 chunk，最后做一个 log-sum-exp reduce 合并局部 softmax。这不是新算法，是把 FA1 的 online softmax 往 KV 维再展开一次，decode 吞吐立刻 2–4×。FlashInfer [6] 把这一套和 paged KV cache、GQA [8] / MQA [9] 的 head 合并、prefix cache、speculative decoding verify kernel 打成一个统一的 C++/Python library，2024Q4 成为 vLLM 默认 attention backend。结论是刺耳的：2026 年自己写推理 attention kernel 的 ROI 基本为负——除非你在做一个 FlashInfer 还没覆盖的新 mask 形状。另一个常被低估的事实是 GQA/MQA 已经从 '优化' 变成 '假设'：FA3 decode path 和 FlashInfer 的默认配置都假设 KV head ≤ 1/4 query head；在这个前提下 KV cache 带宽不再是瓶颈，attention kernel 的优化空间更多让给 prefix cache 和 speculative decoding 的 verify 阶段。

§3 FlexAttention：变体也要 FA 级性能

在 FlexAttention [4] 出现前，生态的尴尬是：FA 只覆盖 causal、full、fixed-window 几种 mask；一旦要 ALiBi、sliding window 变体、MoA、prompt soft mask、document mask，就只能选 (a) 回到 naive PyTorch attention 损失 10–30× 吞吐，(b) 写几百行 CUDA。[10] 之类工作证明了 'FA on variant' 在 kernel 层可行，但每一种变体都要一份手写 kernel，工程成本不可持续。FlexAttention 的做法是：attention score 的修饰（ALiBi 加 bias、log-scale、prompt mask 加法）写成 `score_mod(score, b, h, qi​dx, kv_idx) -> score`；mask 写成 `mask_mod` 返回 bool。torch.compile 把这两个 Python callable inline 到一个 FA 风格 Triton fused kernel 里，第一次调用有编译开销，之后和手写 kernel 性能 gap 只在 5–15%。实测 ALiBi 拿到 FA2 的 92%、sliding window 89%、document mask 85%。底层靠的是 Triton [7] 的 block-level DSL——没有 Triton 的自动 shared-memory 管理和 swizzle，FlexAttention 这种 'Python callable in, fused CUDA kernel out' 的路径不可能成立。组合起来的含义：2026 年变体 attention 的 default 是 FlexAttention，FA3 手写 CUDA 只在两种情况下出场——FlexAttention 还没有的 mask 模式，或变体在 production critical path 上并且那 10% 性能差会烧钱。

§4 FP8 数值现实与 'replace attention' 阵营

两个常被简化的话题值得并排讨论。一是 FP8 attention 的数值现实。FA3 FP8 [3] 在 H100 上对 10B 级 LM pretrain 的 per-step loss 偏差 <0.1%，看起来无风险；但 [17] 的系统 audit 指出某些 rescale 顺序下 BF16 FA 与 reference 的偏差能放大到 1e-3 量级，FP8 下会进一步被 E4M3 的动态范围裁剪。[15] 的 E4M3/E5M2 spec 要求前向用 E4M3（精度高）、反向用 E5M2（动态范围大），FA3 FP8 的 block-wise scaling 照做。生产经验：FP8 训练必须跑 2–5B step 的 BF16 对照，并监控每层 QK 最大 absolute value——这些就是 massive activations [18] 会爆 FP8 范围的位置。二是 SSM / 线性注意力是否值得替换 attention。RetNet [13] 在 2023 年提出三重兼得；[14] 在 8B/3.5T 规模的系统实验给了更冷静的答案：短 context LM loss 相当，但 in-context learning 和 retrieval 差 2–5 pp；hybrid 架构（1/6 attention 层）能补回差距。意思是——'完全替换 attention' 在 2026 的证据密度还不够，但 'attention 层比例从 100% 降到 16%' 已是可验证的现实。所以 attention kernel 优化的收益在 hybrid 里并未消失，只是从 100% 的层打到 ~16% 的层；FA3 + FlashInfer 反而在 hybrid 上更有价值，因为那 16% 的层必须是 full-precision long-context attention。

研究立场对比

阵营 A：FA 系列就是 attention 工程的终点

立场 — attention kernel 的主要答案已经出在 FA1/2/3 [1][2][3]，后续工作只是移植到新硬件；变体和 serving 都只是衍生问题。

证据：[1][2][3]

反方 — 这忽略了 FlexAttention [4] 和 FlashInfer [6] 带来的工具化红利——写 kernel 的人变少，写 variant 的人变多，生态重心已转移。

判词 — 一个更务实的读法：FA 系列是 attention kernel 的基座，但把它当终点会错过 2024–2026 的真正增量——工具化和 serving-native。

阵营 B：Triton / FlexAttention 才是真正的革命

立场 — 真正的转折是 kernel 作者门槛从博士 3 个月降到工程师 1 周 [7][4]；FA3 的 CUDA 内核相对普通工程师不可维护，长期会被 Triton 路径蚕食。

证据：[7][4][10]

反方 — FlexAttention 在 FA3 的 FP8 + warp-specialization 特性面前仍有 5–15% 性能差 [3]；在 frontier training 的临界 batch 上这 10% 就是钱。

判词 — 结论层面的建议：frontier pretraining 的核心 attention 层保留 FA3 手写 kernel，其他 80% 场景交给 FlexAttention——这两条路是互补而非竞争。

阵营 C：attention 本身该被替换（SSM / 线性 RNN）

立场 — 继续优化 O(L²) 的 attention 在 long-context 时代是错位；RetNet [13]、Mamba [14] 已在短 context LM loss 上追平且拥有 O(L) 推理。

证据：[13][14]

反方 — [14] 自己的 8B 实验显示 retrieval / ICL 仍差 2–5 pp；hybrid（1/6 attention 层）才是稳态，而不是纯 SSM。

判词 — 一个更稳的定位：attention 会在 2026 的 hybrid 架构里从 100% 降到 ~16%，但那 16% 的层会更依赖 FA3 + FlashInfer，attention kernel 优化的 ROI 没有消失，只是浓缩。

阵营 D：FA 是 NVIDIA 锁定的具象化

立场 — FA3 [3] 的核心依赖 Hopper 专属指令（TMA、warp-specialization、FP8 tensor core）[16]，在 AMD / 国产芯片上无可移植路径。

证据：[3][16]

反方 — FA1/2 在 AMD MI300 和若干国产卡上已有工程团队移植，Triton 的 AMD backend 也在 2024 成熟；锁定集中在 FA3 FP8 一代，上游 FA2 + FlexAttention 的可移植性明显更好。

判词 — 操作建议：在 non-NVIDIA 硬件上以 FA2 + FlexAttention 为默认路径，FP8 训练 defer 到厂商 kernel 追上——不要把 FA3 FP8 的 2× 当作 attention 的默认可用速度。

实践要点

一份 2026 可执行清单（每条都绑到具体证据）：
1. H100/H200 训练，BF16/FP16 attention 默认 FA3 [3]；A100/H20 默认 FA2 [2]。不要在新项目里跑 FA1。
2. FP8 训练必须跑 2–5B step 的 BF16 对照并监控每层 max|QK| —— [17] 已经证明某些 rescale 路径会放大误差，[18] 的 massive activations 正是 FP8 最先爆的位置。
3. 推理 serving 不要自己写 attention kernel，直接接 FlashInfer [6] 或其所集成的框架（vLLM/TensorRT-LLM）；rolling your own 的 ROI 在 2026 基本为负。
4. 变体（ALiBi、SWA、MoA、document mask、soft prompt mask）默认 FlexAttention [4]；只在 production critical path 并且 5–15% 性能差会烧钱时才回退到手写 CUDA。
5. 长上下文训练把 FA3 与 Ring Attention [11] 组合使用，而不是二选一；FA3 解决单卡 utilization，Ring Attention 解决跨卡 seq 维并行。
6. head_dim 选 64/96/128/192/256，这些有 pre-built 快路径；非标 head_dim 先用 FlexAttention 测一轮再决定是否手写。
7. 不要把 'SSM 替换 attention' 写进 2026 路线图：[14] 的 retrieval/ICL 差距还在；真要下注选 hybrid（1/6 attention 层），并且那 16% 层仍由 FA3 + FlashInfer 承担。
8. 在 AMD / 国产硬件上不要承诺 FA3 FP8 的 2×；默认 FA2 + FlexAttention，把 FP8 当成厂商 kernel 成熟后再启用的 option。

悬而未决的问题

  • Q1.FP8 attention 在 trillion-token、100B+ 规模下的长期稳定性仍缺公开数据；[17] 的 audit 停在 10B 级。
  • Q2.FlexAttention 相对手写 FA3 的 5–15% 性能差能否在 2026 年靠 compiler pass 压到 <3%？这决定了 FA3 CUDA 手写是否还有新项目。
  • Q3.hybrid 架构里 attention 层的最优比例（1/8 vs 1/6 vs 1/4）和其中 long-context attention 的 kernel 策略尚无定论。
  • Q4.非 NVIDIA 硬件（AMD MI300X/MI350、Trainium、国产卡）上 FA3 级别功能的可移植性，特别是 FP8 + async TMA 等效原语的覆盖。
  • Q5.speculative decoding 与 FlashInfer 的 verify kernel 怎么和长 prefix cache 共存，在 batch mixing（多请求不同 prefix 长度）下的吞吐收益仍无开源对照。
  • Q6.FlexAttention 的 Python callable 组合语义（多个 score_mod 叠加、mask_mod 嵌套）在 JIT 编译稳定性与错误诊断上仍是工程痛点。
