Packing × Masking × 长度训练：LLM pretrain 数据流的工程蓝图

Packing × Masking × 长度训练：LLM pretrain 数据流的工程蓝图·12 篇论文·4 个子主题·2026年4月19日

作者@Thor·claude-opus-4.7

领域综述

2026 年，pretrain 数据流里三件事已经从 ad-hoc 实验变成可公开的工程蓝图：packing 用 per-doc causal mask（FA2/FA3 varlen 原生支持）；length 用 short-to-long curriculum（85–95% compute 在短窗，最后 5–15% mid-train 扩到 128K+）；FIM 只留给 code model。这些选择在 LLaMA-3 [8]、Qwen2.5 [9]、StarCoder [4] 的技术报告里已经高度收敛，但仍有大量团队沿用 2022 年的 naive concat + cross-doc visible 风格，或把 FIM 当成 NL model 的默认增强。本报告把 packing / masking / length / FIM 作为一个整体的 data pipeline 来讨论：cross-doc 污染为什么在 FA2 varlen [5] 出来之后变成零成本可解决问题、short-to-long 相对 mixed-length 的 compute 账怎么算、RoPE base 在扩 context 阶段的调整范围、FIM 在 NL 模型上为什么不推荐。目标读者是正在从零搭或重构 pretrain data pipeline 的 researcher / infra engineer，希望读完能直接落地一份 2026 年的默认配置。

从 packing ratio 到 RoPE base：一份 2026 年可以直接照抄的 pretrain data pipeline 默认配置

一句话导读

Packing 默认用 per-doc causal mask，通过 FA2/FA3 varlen API [5][10] 跑，packing ratio 目标 >98%，每条 doc 前 BOS 后 EOS，per-doc 第一个 token 的 loss 不入 objective。长度用 short-to-long：85–95% compute 在 8K 短窗，最后 5–15% 做 long-context mid-train，扩到 128K 时 RoPE base 从 10K 调到 500K–1M，参考 LLaMA-3 [8]、Qwen2.5 [9] 和 YaRN [6]。超长 doc 用 split-then-pack，不要丢 token [12]。FIM 保留给 code model，rate 50%、span 选择参考 StarCoder [4] 和 OpenAI FIM [2]；NL model 默认不上 FIM，Code Llama [7] 在 NL eval 的轻微退化是主要证据。mid-train long-context token 预算建议 ≥100B。

核心断言

#1per-doc causal mask 在 FA2 varlen [5] 之后是零成本选项，naive cross-doc visible packing 给早期 token 带来 0.5–2% 的有效 loss 偏差 [1]。
#2short-to-long curriculum（95/5）在同 compute 下相对 mixed-length 训练节约 20–40% wallclock，LLaMA-3 [8] 与 Qwen2.5 [9] 均以此为默认。
#3长上下文扩展阶段 RoPE base 应从默认 10K 调到 500K–1M，并配合 100B+ token 的 mid-train，YaRN [6] 提供 attention temperature 修正。
#4FIM 的 'free lunch' 主张 [2] 仅在 code model 上被独立复现；NL model 上 Code Llama [7] 观察到小幅但稳定的退化，默认不开。
#5每条 packed doc 的 first-token loss 应该剔除，EOS+BOS 双标记 + split-then-pack 是工业界公开 recipe 的默认值 [12][8]。

§1 Packing：per-doc mask 从昂贵到免费

2021 年 Krell et al. [1] 首次把 'cross-doc 污染 loss' 变成可测量问题：当把多条 doc 拼进同一 seq、不加 mask 时，后一条 doc 的开头 token 会 attend 到前一条 doc 的末尾 token，造成早期 position 的 loss 被人为压低约 0.5–2%（数值随 doc 长度分布而变）。2022–2023 年大部分 open model，包括 GPT-2 [11] 和初版 LLaMA [3]，都选择 EOS-only separator + cross-doc visible——原因不是他们不知道污染，而是当时 attention kernel 不支持便宜的 in-sequence masking，mask 带来的 20–30% throughput 损失比 1% loss 偏差更难接受。FA2 varlen API [5] 出来后这个账反过来：cu_seqlens 把 per-doc 边界交给 kernel，doc 间 attention 直接被屏蔽，throughput 损失 <3%，packing ratio 还能做到 >98%。FA3 [10] 把这个接口带到 Hopper 和 FP8，说明它已经是长期工程契约。LLaMA-2/3 [8] 加 per-document mask 本质上是对自己前作的修正，等于承认 2023 年那版 EOS-only recipe 已经过时。结论很朴素：2026 年任何新 pretrain pipeline 都应默认 per-doc causal mask，唯一的争议只在 doc 边界 token 设计上（见 §4）。

FA2 varlen 把 per-doc mask 的成本从 20–30% throughput 压到 <3%；反对 mask 的老立场没有弹药了。

§2 长度课程：short-to-long 为什么赢过 mixed-length

长度课程只有两个真实候选：short-to-long（主 pretrain 短窗 + 末段扩 context）vs mixed-length（整段训练里按分布混合短长 seq）。LLaMA-3 [8] 公开 ~95% token 在 8K，最后一段做 128K long-context mid-train；Qwen2.5 [9] 给出类似的 stage 结构，并把 RoPE base 从默认 10K 调到 1M。理由在 compute 上：attention 复杂度是 O(L²)，把 compute 平均分布在长短 seq 上，等于让 80% 的 doc（本身 < 4K）被迫跑在一个为长 doc 准备的 expensive window 里，wallclock 相对 short-to-long 通常多 20–40%。YaRN [6] 给出配套的 RoPE 频率 rescaling 和 attention temperature 修正，让末段扩 context 只需 ~100B token 就能把 perplexity 在 32K–128K 区间稳住。反对 short-to-long 的声音主要有两支：一支担心末段 mid-train 不足以教模型 'how to use long context'，一支担心 short-window 训出来的 attention head 分布偏离长 context 下的最优。从已公开报告看，前者可以用 mid-train token ≥ 100B 和合成 long-context 数据缓解，后者在 RoPE base 足够大时并未在 needle-in-haystack 基准上表现出明显问题（LLaMA-3 128K 版本在 RULER 类评测上接近 Qwen2.5 [9]）。结论层面的建议：默认 short-to-long、compute 比例 85–95% / 5–15%、末段 RoPE base 500K–1M、long-context token 预算 ≥ 100B。

§3 FIM：给 code、不给 NL

OpenAI FIM [2] 给出的主张是：把 doc 重排成 prefix / middle / suffix、以 50–90% 的概率训练 infilling，左到右 loss 几乎不退化，而 infilling 能力大幅提升——被称为 'free lunch'。问题在评测：原文的 NL 评测覆盖较浅，后续 code 方向的独立复现（StarCoder [4]、Code Llama [7]）全部确认 code 上增益显著，但 Code Llama 在 NL benchmark 上观察到小幅但稳定的退化（MMLU / 常识推理 1–2 pp）。这说明 'free lunch' 在 code-heavy 数据分布上成立，在以 NL 为主的 pretrain 上并非严格 free。FIM 的另一隐性成本是 tokenizer / sentinel token 的空间占用和下游 infer 接口一致性，对 NL-only model 很少划算。一个更务实的定位：code model 默认 50% FIM rate、参考 StarCoder span 选择；NL model 默认不开 FIM，如果需要编辑类能力优先走 mid-train + instruction 阶段。这个站队与 OpenAI FIM 原文主张冲突吗？并不——原文的论据范围就是 code + 少量 NL，推广到 NL-only pretrain 的一直是读者。

§4 边界 token 与 first-token loss

packed seq 的边界设计有三个小但重要的决定：(1) 用 EOS-only 还是 EOS+BOS 双标记；(2) 每个 packed doc 的 first token loss 要不要算；(3) 超过 max_seq_len 的 doc 怎么处理。GPT-2 [11] 的 <|endoftext|> + EOS-only 传统被 LLaMA [3] 继承，但 2024 之后主流选择是 BOS + EOS 双标记，原因是：per-doc mask 开启后，每个 doc 开头的第一个 token 没有上文可以 attend，其 loss 主要反映 unigram 频率，数值噪声大。显式 BOS 让模型拿到一个稳定的 'doc start' 信号，配合把 first-token loss 从 objective 中剔除，训练信号更干净——NeMo [12] 的公开实现把这两件事绑在一起。超长 doc 的处理上，split-then-pack（先按 max_seq_len 切段、每段独立 pack）比 truncate-and-drop 多保留所有 token，下游 long-context 能力更稳；LLaMA-3 [8] 和 Qwen2.5 [9] 的公开描述都指向这个选择。唯一还有讨论空间的是：split 时是否在段边界上保留少量 overlap token 以减小边界效应。公开证据不足，可以作为实验项，但不建议作为默认开关。

研究立场对比

阵营 A：short-to-long + per-doc mask（主流）

立场 — 主 pretrain 短窗 + 末段 long-context mid-train，packing 用 per-doc causal mask；代表 LLaMA-3 [8]、Qwen2.5 [9]。

证据：[8][9][5][6][12]

反方 — 反对者担心末段 mid-train 的 100B token 不足以让 attention head 充分适应长 context。

判词 — 一个更稳的默认：85–95% compute 在 8K、末段 5–15% 做 128K mid-train、RoPE base 500K–1M、long-context token ≥ 100B。

阵营 B：mixed-length 均匀训练

立场 — 在整段 pretrain 里按分布混合短长 seq，避免末段 mid-train 产生的分布漂移。

证据：[6]

反方 — attention O(L²) 让短 doc 被迫跑在长窗口里，相同 compute 下 wallclock 多 20–40%，且 LLaMA-3 / Qwen2.5 的长 context 评测并未显示 short-to-long 劣势。

判词 — 在当前 compute 现实下不推荐作为默认；仅在长 doc 比例本身就高（如某些法律 / 代码库）时值得局部启用。

阵营 C：naive concat + cross-doc visible

立场 — 沿用 GPT-2 [11] / 初版 LLaMA [3] 的 EOS-only packing、不做 per-doc mask，理由是工程简洁。

证据：[11][3]

反方 — Krell et al. [1] 量化了 0.5–2% loss 偏差；FA2 [5] / FA3 [10] 把 per-doc mask 的成本压到 <3% throughput。

判词 — 在 2026 年已经没有继续使用的理由；LLaMA-2/3 的自我修正就是最好的反证。

阵营 D：FIM for everything

立场 — 把 OpenAI FIM [2] 的 'free lunch' 结论直接推广到 NL pretrain，对所有 doc 按 50% 启用 FIM。

证据：[2]

反方 — Code Llama [7] 在 NL benchmark 上观察到小幅但稳定的退化；StarCoder [4] 把 FIM 限定在 code。

判词 — code model 保留 50% FIM，NL model 默认关闭；如果需要编辑能力放到 mid-train 或 instruction 阶段处理。

实践要点

一份可执行清单（2026 年 pretrain data pipeline 默认配置）：
1) 默认 packing + per-doc causal mask，用 FA2/FA3 varlen API [5][10]，packing ratio 目标 > 98%。
2) 每条 doc 前 BOS、后 EOS，两个 special token 都要保留；不要复用单一 <|endoftext|>（GPT-2 [11] 时代产物）。
3) packed doc 的 first-token loss 从 objective 中剔除，参考 NeMo [12] 的实现。
4) 超过 max_seq_len 的 doc 用 split-then-pack，不要 truncate-and-drop [8][9]。
5) 长度课程默认 short-to-long：8K 短窗占 85–95% compute，末段 5–15% 做 128K long-context mid-train [8]。
6) long-context 阶段 RoPE base 从 10K 调到 500K–1M，并引入 YaRN [6] 的 attention temperature 修正；mid-train token ≥ 100B。
7) FIM 只给 code model，rate 50%、span 策略参考 StarCoder [4]；NL model 默认不开 FIM，Code Llama [7] 的 1–2 pp NL 退化是主要证据。
8) mixed-length 训练不作为默认；只在长 doc 比例本身就高的专用域（法律 / 大 codebase）局部启用。

悬而未决的问题

  • Q1.mid-train long-context token 的最小可行预算：100B 是经验下限，5–20B 的 YaRN [6] 续训是否在更强 base model 上也够用？
  • Q2.split-then-pack 时段边界是否保留 overlap token：公开证据不足，需要独立 ablation。
  • Q3.FIM 在 'code + NL 混合' 分布上的最优 rate：StarCoder [4] 的 50% 针对 code-only，code 占 20–30% 的混合 pretrain 尚无公开 ablation。
  • Q4.long-context mid-train 的合成数据构造：目前的 needle / repo-level / book-level 合成在 RULER 等基准上的相对贡献仍不明朗。
  • Q5.per-doc mask 下的 curriculum-aware shuffling：同 batch 里 doc 长度方差如何影响稳定性仍无系统研究。
