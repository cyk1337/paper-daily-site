Code 浓度 × Pretrain：reasoning 迁移、御三家实践与 20% sweet spot

Code density in pretrain: reasoning transfer, big-three practices, and the 20% sweet spot·15 篇论文·4 个子主题·2026年4月19日

作者@Thor·claude-opus-4.7

领域综述

2024–2026 的 frontier pretrain 在 code 占比这件事上出现了明显收敛：通用 model 集中在 15–25% 区间，中位数约 20%；code specialist 则稳定在 50–90%。这一区间不是品味问题，而是 DeepSeek-Coder [1]、Code Llama [2]、Orbay et al. [3] 的 ratio ablation、Aghajanyan et al. [4] 的 synergy/interference 相图，以及 Gadre et al. [5] 的 downstream variance 数据共同逼出来的结论。机制上有两条独立线索：token entropy——code 的可压缩性高，相当于更低的 effective LR，解释了 downstream noise 更低的现象；以及 induction head——Olsson et al. [8] 把 ICL 相变定位在 induction circuit 上，而 code 的 repeat-token 密度正好是这种 circuit 的天然食物。工程侧 repo-level packing [6] 与 FIM rate 0.5–0.9 [7] 已是默认配置。这篇 report 的目标不是讲 'code 很重要'，而是把 <15% / 15–25% / 25–40% / >40% 四段的 NL / reasoning / code 代价量化到可操作程度，并区分 from-scratch 与 continual 两条截然不同的路径——后者并不自动 catastrophic [2]。

为什么 2026 frontier 的 code 比例收敛到 ~20%——以及更高 / 更低都有哪些隐性代价

一句话导读

到 2026，frontier generalist 的 code 占比收敛在 15–25%（默认 20%），specialist 在 50–90%。三组独立 ratio ablation（DeepSeek-Coder [1]、Code Llama [2]、Orbay et al. [3]）形状一致：<15% reasoning under-trained，>40% 才看到 NL 退化。机制上 Aghajanyan et al. [4] 的 synergy/interference 相图与 Gadre et al. [5] 的 downstream-variance 数据指向同一解释——code token entropy 低，等效于更低的 effective LR；Olsson et al. [8] 把这条线索下钻到 induction head circuit。工程侧 repo-level packing [6]、FIM rate 0.5–0.9 [7]、结构 token [12] 已是 default。Code Llama [2] 反驳了 'continual 一定 catastrophic' 的 folklore，但 Ma et al. [14] 同时指出只在 annealing 加 code 的迁移远弱于全程混入——from-scratch 与 continual 不可混为一谈。

核心断言

#1通用 model 的 code ratio 默认 20%，区间 15–25%；<15% 是 under-trained reasoning，>30% 才进入 NL 风险区（[1][2][4]）。
#2Code specialist 合理区间是 50–90%，对应 MMLU 相对通用 base 下降 6–9 pp；不要假装 NL 不会退，要用 post-training 补回（[1]）。
#3Code 加的是 reasoning 里面 '结构化 / 多步' 那部分，不是 commonsense 全域；Petty et al. [15] 的 controlled 实验表明数学 +2–5 pp、commonsense 不显著。
#4Continual 不是 catastrophic 的充分条件：Code Llama [2] 500B code continual 让 MMLU 只掉 <1 pp；但 Ma et al. [14] 指出 annealing-only 加 code 的迁移远弱于全程混入。
#5Repo-level packing [6] + FIM 0.5–0.9 [7] + 结构 token [12] 已是 2026 default；少做一条，cross-file 补全掉 5–10 pp。

§1 证据带的形状：三组独立 ablation 为什么指向同一个区间

把 DeepSeek-Coder [1]、Code Llama [2]、Orbay et al. [3] 的 ratio 曲线叠到同一张图，结论是稳的：code 占比从 0 到 ~15% 之间，reasoning / code benchmark 都在陡涨、NL 几乎无损；15–25% 是一条平台，reasoning 继续小幅上升、NL 依旧稳；到 30% 附近 NL benchmark 开始出现可测下滑（MMLU / Hellaswag 级别 1–2 pp），到 40% 以上掉得更快。DeepSeek-Coder [1] 在 87% 处给出的 specialist 数据点最极端——HumanEval 高 20+ pp，但 MMLU 相对同 scale 通用 base 低 6–9 pp，两端都很干净。Code Llama [2] 是另一个极端的对照：在 LLaMA-2 上 continual 500B code，MMLU 只掉 <1 pp 而 HumanEval +25 pp——说明这条曲线在 continual setting 下被 base 的 NL 知识'锚住'，退化幅度显著小于 from-scratch。Orbay et al. [3] 从第三个角度进来：code-only pretrain 的 compute-optimal token:param 与 NL 一样接近 ~20，拟合范围内没 saturation——这意味着 2026 frontier 更容易卡在 'code 不够' 而非 'code 过多'。Llama 3 [9] 的 data-mix annealing 在 15T 后期上调 code 比例，是 frontier lab 实际落在 15–25% 的旁证。三条线索都在同一形状上收敛，这个区间不是品味问题。

§2 机制：token entropy、effective LR 与 induction head

为什么曲线是这个形状？两条互相独立又彼此一致的机制线索值得写清楚。第一条是 token entropy。code 的 identifier 重复、模板化的语法、严格的括号配对使其 bits/token 显著低于 web——Gadre et al. [5] 的 downstream-variance 数据最直观：同 loss 下 code domain 的 downstream noise 比 web 低约 30%。这不是 'code 更简单'，而是 code 的 gradient 信号更稳定，等效于一个更低的 effective LR。一个直接推论：在固定 LR schedule 下，code 的加入会把整体训练动力学往 '更平滑的 curvature' 方向拉——解释了为什么 15–25% 区间能同时不伤 NL 又帮 reasoning。第二条是 induction head。Olsson et al. [8] 把 in-context learning 的相变定位在 induction circuit 出现的那一刻；code 的 repeat-token 密度、变量复用、import 依赖都是 induction 的天然训练信号。从这个角度看，code 不是 '多给一种语言'，而是 '给模型一批高信噪比的 pattern-copy 样本'——这也是为什么 Madaan et al. [13] 观察到 code-pretrained LM 在结构化 commonsense 上反而赢 NL-pretrained。Aghajanyan et al. [4] 的 synergy/interference 相图给这两条线索提供一个定量外壳：<10% synergy dominant、>40% interference dominant——正好框住 code ratio 的 sweet spot。但要注意一个边界条件：Petty et al. [15] 的 controlled 实验显示 code 的正向迁移集中在数学与结构化任务（+2–5 pp），开放域 commonsense 上效应不显著——code 不是万能 reasoning 药。

Code 在 pretrain 里的作用不是'教一门语言'，而是'喂一批高信噪比的 pattern-copy 样本'。

§3 两条路径：from-scratch 与 continual 不可混为一谈

关于 code-heavy continual 'necessarily catastrophic' 的说法 2026 还在流传，但证据并不站在它一边。Code Llama [2] 在 LLaMA-2 上 500B code continual，MMLU 下降 <1 pp，HumanEval +25 pp——如果 catastrophic forgetting 是定律，它就不该发生。可能的机制解释有两条：其一，base model 已经把 NL representation 训到相对稳定的区域，后续 high-code 梯度更像在 fine-tune feature；其二，Code Llama 保留了 8% 的 NL replay，在 replay-based continual 文献下这足以锚定高层 representation。另一面不能忽略 Ma et al. [14] 的发现：code 'what stage to add' 是独立维度——全程混入的 reasoning 迁移最强，只在 annealing 阶段加则明显弱。把这两条放在一起，结论是：continual 不是一个开关，而是一条两维的决策线——持续时长 × NL replay 比例。反过来，from-scratch 的曲线更锐利：DeepSeek-Coder [1] 的 87% 数据点上 NL 退化是 hardcoded 在 data mixture 里的，无法 post hoc 弥补，只能在 post-training 阶段通过 SFT + NL-heavy RLHF 补回 4–6 pp。工程上这意味着两条路径应该走不同的评估门槛：from-scratch 的 MMLU 要在 pretrain 中期就 gate，一旦掉出可接受带立刻压 code 比例；continual 的 gate 点在 replay 比例与 schedule 端点——只要 NL replay ≥ 5% 且 end-LR 不过热，掉幅通常可控。

§4 工程栈：repo-packing、FIM 与结构 token 的默认值

到 2026，code pretrain 的工程栈几条已经是 default——不做或做错都有可量化代价。第一条是 repo-level packing：按 import / include 依赖图聚簇把一个 repo 的文件拼到同一 sequence，而不是随机 concat。Shi et al. [6] 把这个做法理论化，cross-file completion +5–10 pp；StarCoder [12] 则把 `<|file_sep|>` / `<|repo_name|>` 结构 token 固化成 recipe——没有这些 token，packing 的收益会掉一半，因为模型无法区分 'next file starts' 与 'code block continues'。第二条是 FIM。Bavarian et al. [7] 证明 FIM rate 最高到 0.9 对 left-to-right loss 几乎零代价，而 infilling 能力直接拉满；specialist 用 0.5–0.9，generalist 用 0.1–0.3，再保守也不应是 0。第三条是 dedup + license filter。The Stack [11] 把 pipeline 打开给社区：近似 dedup 在 HumanEval 上给出 1–3 pp 增益，license filter 不仅合规还剔除了大量低质量 fork copy。第四条被低估：语言分布。Python 30–40%、C/C++ + JS/TS 20–30%、其它 30–40% 是目前 open recipe 的共同画像；不要把低资源语言压到 <1%——它们是 few-shot long tail benchmark 的来源，也是 induction head 拓宽到非 Python domain 的必要食物。反过来，如果四条工程项全省略——随机 concat、FIM rate 0、无 dedup、Python 90%——和做满的 baseline 差距经常在 10–15 pp 量级，这个代价在任何 data-mix 讨论之上。

§5 code 到底让什么更强：把 'reasoning' 拆开看

'加 code 让 reasoning 更强' 这句话如果不拆就是口号。把 reasoning 拆成结构化多步、数学、commonsense、长距离 retrieval 四段会更诚实。Madaan et al. [13] 在结构化 commonsense 上观察到 code-pretrained LM 同 scale 领先 NL-pretrained 5–10 pp，这一段是最稳的正向迁移。数学方面 Petty et al. [15] 的 controlled 实验固定 compute 与 token 总量、只动 code-mix，正向效应 +2–5 pp，但同样的实验显示开放域 commonsense 上效应不显著——这直接削弱 'code 是全域 reasoning 银弹' 的强命题。长距离 retrieval 则落在 repo-level packing 上：Shi et al. [6] 的 cross-file gain 本质是模型学会了在长 context 里 trace 符号定义，这与 Chen et al. [10] 最初在 Codex 观察到的长 context code completion 增益是同一件事。把这四段合起来：code 加的是 reasoning 里面 '结构化 / 多步 / 长距离' 那部分，不是 commonsense 全域——这解释了为什么 15–25% 区间能在 reasoning benchmark 上稳定拿到 2–5 pp 却不伤 NL。反过来也提示了 post-training 的分工：SFT 补 commonsense 与 open-ended，RLHF 补 calibration 与 safety，pretrain 的 code share 负责结构化 reasoning 的地基——三件事不要彼此替代。

研究立场对比

阵营 A：code 越多越好，通用 model 也该推到 40% 以上

立场 — 既然 code 帮 reasoning，且 Orbay et al. [3] 没看到 saturation，那就继续加——40% 甚至 60% 对 generalist 也应该更好。

证据：[3][13]

反方 — Aghajanyan et al. [4] 的相图显示 >40% 进入 interference 区；DeepSeek-Coder [1] 87% 的 MMLU 缺口 6–9 pp 是真金白银；Petty et al. [15] 也证实 code 的迁移不覆盖 commonsense 全域。

判词 — 结论层面的建议：generalist 不应越过 30%，40% 之上是 specialist 领域。想用 reasoning 增益，靠更长 training 与更好 quality filter，不是更高比例。

阵营 B：code 的收益只是 regularisation / 更低 effective LR

立场 — Gadre et al. [5] 显示 code downstream variance 低，说明 code 只是起 'lower effective LR' 的作用——任何稳定数据源都可以替代。

证据：[5]

反方 — Olsson et al. [8] 提供了 circuit 级证据：code 的 repeat-token 结构是 induction head 的特异食物，不是一般低 entropy 数据可以替代的；Madaan et al. [13] 的结构化 commonsense 增益也无法用 'effective LR' 解释。

判词 — 一条更稳的读法：code 同时提供 low-entropy regularisation 与 induction-friendly pattern——两个机制叠加，用其它数据源无法等效替换。

阵营 C：code 应当 <10% 以保护 NL

立场 — code 抢 NL 的 capacity，generalist 应尽量压到 5–10%；NL 能力是产品线的根，绝不能被牺牲。

证据：[4]

反方 — 三组独立 ratio ablation [1][2][3] 都显示 <15% 时 reasoning under-trained；Llama 3 [9] 在 15T 后期反而抬高 code 占比——'<10% 保 NL' 与 frontier 实际选择相反。

判词 — 一条更稳的读法：15–25% 既不伤 NL 也拿到 reasoning 收益，是 'NL 保护者' 应该选择的区间，而不是 <10%。

阵营 D：code 能力必须 from-scratch，continual 是 doomed

立场 — continual code-heavy 会 catastrophic forgetting NL，generalist→specialist 的唯一干净路径是 from-scratch。

证据：[1]

反方 — Code Llama [2] 500B continual 让 MMLU 只掉 <1 pp——证据本身就反例；Ma et al. [14] 也显示只要保留 NL replay 并让 code 全程参与，continual 的迁移可达 from-scratch 80%+。

判词 — 一条更稳的读法：continual 不是失败路径，而是一条两维决策线——持续时长 × NL replay 比例；设计好这两个旋钮，continual 的性价比往往高于重新 from-scratch。

实践要点

一条可执行的清单（2026，同时面向 generalist 与 specialist 两条线）：1) 通用 model code ratio 设 15–25%，默认 20%；<15% 是 under-trained reasoning，>30% 进 NL 风险区（[1][2][4]）。2) Specialist 设 50–90%，直接 accept 6–9 pp 的 MMLU 缺口，post-training 阶段用 SFT + NL-heavy RLHF 补 4–6 pp（[1]）。3) 语言分布 Python 30–40% / C/C++ + JS/TS 20–30% / 其它 30–40%，低资源语言别压到 <1%——它们是 few-shot long tail 与 induction 拓宽的必要食物（[12][8]）。4) Repo-level packing 按依赖图排序 + `<|file_sep|>` 结构 token 强制；省略它 FIM 与 cross-file 同时掉 5–10 pp（[6][12]）。5) FIM rate specialist 0.5–0.9、generalist 0.1–0.3，不要设成 0——Bavarian et al. [7] 证明它近似 free。6) Dedup + license filter 做满，near-dup 值 1–3 pp HumanEval（[11]）。7) Continual 路径不是 catastrophic 的同义词，但必须保 ≥5% NL replay 且 code 全程参与；annealing-only 加 code 是弱鸡（[2][14]）。8) 不要声称 code 提升全域 reasoning——Petty et al. [15] 已证实开放 commonsense 上效应不显著；要承认 code 的作用边界是结构化 / 多步 / 长距离。

悬而未决的问题

  • Q1.在 >10T token 的 over-training regime 下，code ratio 是否需要与 token-compute 一起被重新校准？Orbay et al. [3] 的拟合范围没覆盖到这一段。
  • Q2.Synthetic code（LLM 自生成、编译器 trace 扩展等）应计入 code ratio 吗？它与 human code 在 induction-head 训练信号上是否等价？
  • Q3.多模态 base（image + text + code）下 Aghajanyan et al. [4] 的 synergy/interference 相图是否仍然成立？code 的 effective LR 效应可能被 vision token 的高 entropy 抵消。
  • Q4.continual path 的 NL replay 比例与 end-LR 之间的交互还缺 controlled 数据——Code Llama [2] 只给出了一个 operating point。
  • Q5.Code quality filter（static analysis、compile-pass、test-pass）相对 quantity 的 trade-off 曲线仍 open；现有 recipe 多是'pipeline 给定，看跑出来多少'而非系统性 ablation。
