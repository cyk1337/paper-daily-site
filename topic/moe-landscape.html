MoE 全谱对比 2026：路由 / 负载均衡 / 专家结构与 DeepSeek 模板的胜出

MoE Landscape 2026: routing, load balancing, expert structure, and why the DeepSeek template won·20 篇论文·4 个子主题·2026年4月19日

作者@Thor·claude-opus-4.7

领域综述

过去六年 MoE 的设计空间看起来杂乱，但把所有工作拆到三条轴上就会发现收敛轨迹非常清晰：routing 从 token-choice top-k 走到 expert-choice 再走到 aux-loss-free bias 调整；load balancing 从 GShard 的 aux loss 走到 ST-MoE 的 router z-loss，再到 DeepSeek-V3 用 bias EMA 完全替掉 aux loss；expert structure 从 GShard/Switch 的 8–64 大专家走到 DeepSeekMoE 的 64–256 fine-grained + 1 shared expert 模板。2024 下半年起，这套 'fine-grained + shared expert + aux-loss-free + z-loss' 组合已经是新训 MoE 的事实 default——Mixtral / DBRX 是上一代保守剖面，DeepSeek-V2/V3 是当前锚点，FLAME-MoE / OpenMoE 是复刻样本。这份 position report 把每个设计选择对应到具体 cite_key、量化收益与失败模式，并在 positions 里点名四个仍未消散的对立阵营：MoE-inevitabilist、dense-ROI-better、expert-choice-first、post-train-revert-to-dense。

路由 / 负载均衡 / 专家结构三条轴上，DeepSeek 模板如何收敛开源 MoE 的设计空间

一句话导读

2026 年的 MoE 设计空间可以用三条轴完整刻画：routing、load balancing、expert structure。每条轴上都发生过一次代际更替——token-choice top-k → expert-choice → aux-loss-free bias 调整 [2][4][10]；aux loss → router z-loss → bias EMA [3][5][11]；8–64 coarse → 64–256 fine-grained + 1 shared [7][8]。DeepSeek-V2/V3 把三条最新选择打包成事实 default，FLAME-MoE / OpenMoE 是复刻样本 [9][DeepSeekAI2024V3][16][12]。新栈直接 clone DeepSeekMoE 模板：fine-grained (64–128) + 1 shared + aux-loss-free + z-loss (1e-3)，前 2000 step 按 dead-expert 调 bias EMA 步长 η。反对阵营有 OLMo 3 的 dense-ROI 路线 [27] 与 upcycling 路线的 scaling law 限制 [14]，但都没能动摇模板。

核心断言

#1同 active 参数预算下，DeepSeekMoE 的 fine-grained + 1 shared 相比 Mixtral-style coarse (8×7B, top-2) 在 MMLU 上 +1.8–3.4 pp [8][7]，这是 2024 年模板切换的首要驱动力。
#2aux-loss-free bias EMA 路由不是 'aux loss 的等价替换'，而是结构上更简单：dead-expert 收敛有 stochastic-approximation 证明 [17]，且公开训练日志显示 token drop rate 可稳在 <0.5% [11][10]。
#3expert-choice routing [4] 在 encoder-only 和 prefill 上仍有优势，但 decoder-only autoregressive 推理里无法保留 causal 顺序——这是它在 2024+ 开源 MoE 中没进入事实 default 的硬约束。
#4dense→MoE upcycling 只在 dense checkpoint 已经 token-rich 时划算，有效 token 等效系数 0.4–0.6× [14]；低 token dense 直接 upcycle 的边际收益经常为负，OLMo 3 / Olmo2 坚持 dense 路线有其 ROI 基础 [27][26]。
#5router z-loss [5] 与 aux loss 正交，解决的是 logit overflow 而非负载塌缩——这是 DeepSeek-V3 去掉 aux loss 后仍保留 z-loss=1e-3 的原因 [11]。

§1 三条轴：六年 MoE 设计空间的收敛结构

2019–2026 所有 MoE 论文都可以拆到三条轴上：routing（谁挑谁）、load balancing（怎么让专家别塌缩）、expert structure（专家多大、要不要 shared）。每条轴都发生过一次代际更替。Routing：Shazeer et al. [1] 给出 top-k + aux loss 原型，Lepikhin et al. [2] 把它固化为生产模板（top-2 softmax + aux loss + capacity factor），Fedus et al. [3] 砍到 top-1，Zhou et al. [4] 反转为 expert-choose-token 以结构性消除不均衡，Wang et al. [10] 再次反转：不换路由方向、只用 bias EMA 自我纠偏，最后被 DeepSeek-V3 [11] 大规模验证。Load balance：GShard [2] 的 aux loss 永远在污染主 gradient；Zoph et al. [5] 加 router z-loss 解决 logit overflow 这个正交问题；DeepSeek-V3 把 aux loss 彻底移除，只留 z-loss + bias EMA；Qiu et al. [13] 指出即使用 aux loss，其实现细节（micro-batch vs global、是否 detach p、跨 DP 同步与否）常常比强度更重要——这反过来支持了 'bias EMA 更少坑位' 的工程判断。Expert structure：Shazeer / GShard / Switch / Mixtral [1][2][3][7] 都是 8–64 大专家路线；Dai et al. [8] 切到 64+ 并引入 1 个 shared expert，Nguyen et al. [15] 从 identifiability 角度给出共享项的统计学解释。三条轴上的最新选择在 2024 下半年被 DeepSeek-V2/V3 [9][11] 打包为事实 default，FLAME-MoE 与 OpenMoE [16][12] 提供了可复现实证。

routing / load balance / expert structure：三条轴，各一次代际更替，共同指向同一个模板。

§2 模板对比：Mixtral / DBRX / DeepSeek-V2 / V3 / Qwen-MoE 的剖面

把主流开源 MoE 按三条轴拆开，代际分界非常整齐。Mixtral [7] 代表 2024 年初的保守剖面：8 coarse experts、top-2、aux loss、无 shared expert。DBRX 把专家加到 16 但仍是 coarse 路线。DeepSeek-V2 [9] 在 2024 年中一次性换掉两条轴：fine-grained 160 experts + 1 shared + MLA，仍保留 aux loss；DeepSeek-V3 [11] 在 2024 年底把 aux loss 也换掉，换成 bias EMA，256 routed + 1 shared，21B active / 671B total。Qwen-MoE 与 OLMoE 都朝 fine-grained + shared 靠拢。OpenMoE [12] 公开了 routing trace，展示 context-independent routing / drop-towards-end 等 failure mode——这些是任何新 MoE 栈上线前必须核对的工程 check。MegaBlocks [6] 的 block-sparse kernel 使得 fine-grained 路线不再因为 capacity factor 丢 token——这是模板迁移能落地的系统前提。表格见下。

§3 为什么 aux-loss-free 赢了 aux loss

aux loss 的两个结构性问题没法靠调系数绕开：一是 gradient 污染——aux 对 router logit 的梯度和任务损失混在一起，降低 bias EMA 才能做到的 'routing 决策与模型 loss 解耦'；二是实现脆弱，Qiu et al. [13] 把 aux loss 拆到 micro-batch vs global、detach p 与否、cross-DP 同步与否之后，发现 CV 与专家专化度的波动常常超过 aux 强度从 1e-2 到 1e-4 的全扫。bias EMA 用一个 sign-gradient 规则 bi​ ← bi​ + η · sign(target − load_i) 来实现同样的负载均衡，好处是：它只改 bias、不改 router 权重；η 可按 dead-expert count 动态调整；dead-expert 收敛有 stochastic-approximation 证明 [17]。DeepSeek-V3 [11] 公开的训练日志显示 token drop rate 全程 <0.5%、expert usage CV 稳定在 0.12–0.18，换 aux loss 路线在同 setup 下通常 CV≥0.25。需要强调：z-loss 不能一起去掉。z-loss 管的是 router logit 的绝对 norm（避免 fp16 overflow），和 aux loss 的负载均衡是正交问题——这是 DeepSeek-V3 仍保留 z-loss=1e-3 的原因 [5]。相似思路的外推包括 Similarity Preserving Routers [18] 用结构化 router 抑制塌缩，以及 MoE++ [20] 引入 zero-computation expert 做自适应 compute——两者都沿着 'balance 作为结构约束' 这条线。

§4 post-train、upcycling 与 dense-ROI：模板的边界在哪

MoE 模板的胜出并不等于 MoE 吃掉所有场景。三条边界需要说清。一，post-train：SFT/RLHF 阶段 router 的梯度信号变稀，fine-grained MoE 的稳定性低于 dense 的观察在 2024–2025 多个团队口头提到，但公开发表的 matched-quality ablation 仍然稀缺——Chen et al. [23] 与 Koishekenov et al. [24] 走的是 'MoE pretrain + task-specific expert pruning 部署' 的折中路线，可以视为 Camp D 的早期经验证据。二，upcycling：Liew et al. [14] 的 scaling law 给出明确边界——dense checkpoint 需要先 token-rich（通常 > Chinchilla-optimal），upcycle 后的有效 token 等效系数才能落在 0.4–0.6× 区间；token-poor dense 直接 upcycle 常常负收益。Szatkowski et al. [25] 展示了利用激活稀疏做 dense→MoE 转换的另一种路径，但仍在小 scale。三，dense 路线：OLMo 2 / OLMo 3 [26][27] 以 7B/32B dense 全开源家族的方式证明，单机 serve、post-train 稳定性、研究可复现性在某些任务上比 MoE 的 quality-per-active-param 更关键——Camp B 不是在否认 MoE 的 scaling 优势，而是在指出特定 serving 预算下 dense 的 ROI 仍然领先。serving 端 MegaBlocks [6]、ExpertFlow [21]、MoETuner [22] 把 fine-grained MoE 的 serving 成本压下来，但单机 inference 的内存 footprint 依然是 MoE 无法完全跨越的硬约束。

研究立场对比

阵营 A：MoE 是 dense 的必然替代

立场 — 在同 training FLOPs 下 MoE 的 quality/FLOP 有 1.5–2× 优势，fine-grained + shared + aux-loss-free 模板已解决大半工程坑，MoE 即将成为所有 frontier pretrain 的 default [1][8][11][16]。

证据：[1][8][9][11][16][15]

反方 — 忽略 serving 端 memory footprint 与 post-train 稳定性的约束；在 7–13B active 的部署区间 dense 仍然更省 [27][26]。

判词 — 结论层面的建议：在 pretrain + ≥30B active 的目标 serving 栈上，直接采用 MoE 模板；≤13B active 或单机 inference 为主的场景继续用 dense。

阵营 B：dense 路径最终 ROI 更优

立场 — MoE 的 total-param 占用、跨机 all-to-all 通信、post-train 稳定性在很多实际业务下抵消了 quality-per-active-param 优势；fully-open dense 家族在研究可复现、fine-tune 稳定、单机 serve 上更有优势 [27][26]。

证据：[27][26][23][24]

反方 — 对 frontier-scale ≥70B active 的 quality ceiling 没有回应；DeepSeek-V3 在同 FLOPs 下已 overtake 众多 dense 7–70B [11][9]。

判词 — 一个更务实的定位：dense 是 ≤32B 的最佳投产路径；超过这个规模再考虑 MoE，而不是一开始就 MoE-first。

阵营 C：expert-choice / aux-loss-free 是未来

立场 — 把负载均衡做成结构约束而非软惩罚，是 MoE 路由的正确方向；expert-choice [4] 开了头，aux-loss-free bias EMA [10][11][17] 完成收官；后续 similarity-preserving router [18]、AdaMoE [19]、MoE++ [20] 在同一条线上继续推。

证据：[4][10][11][17][18][19][20][13]

反方 — expert-choice 在 autoregressive decoder 下无法保留 causal 顺序 [4]；aux-loss-free 的 bias EMA 步长仍需人工前 2000 step 监控，并非完全免维护。

判词 — 站 C 阵营但承认边界：decoder-only pretrain 走 aux-loss-free bias EMA；expert-choice 适合 encoder-only 或 prefill-heavy 场景。

阵营 D：MoE 只在 pretrain 重要，post-train 应回 dense

立场 — router gradient 稀疏 + 专家分化让 SFT/RLHF 稳定性下降；实践上 task-specific expert pruning [23][24] 或 dense-to-dynamic-k 转换 [25] 在部署时把 MoE 压回近 dense 结构已多次实证。

证据：[23][24][25][14]

反方 — DeepSeek-V3 在 post-train 阶段保留 256 expert 结构且拿到一线 alignment 效果，说明 MoE post-train 并非结构性不可行，只是工程门槛高 [11]。

判词 — 一个更稳的读法：MoE post-train 可行但代价不低；中小团队如果没有完善的 routing 监控栈，部署端走 MoE→dense-expert-pruning 反而更稳。

实践要点

一条可执行的清单（2026 Q2 版本）：
1. 新建 MoE 栈：clone DeepSeekMoE 模板——fine-grained 64–128 routed experts + 1 shared + aux-loss-free bias EMA + router z-loss 1e-3 [8][11][5]。不要再从 Mixtral-style 8 coarse experts 起步，2024 年之后这条路径的边际收益已经被 fine-grained + shared 占掉 2–5 pp [7][9]。
2. 已有 Mixtral-style coarse MoE：优先把专家拆细到 32–64 并加 1 个 shared expert；routing 可以暂不动。ablation 里 fine-grained 这一步的边际收益最高 [8][15]。
3. aux-loss-free 不是零维护。bias EMA 步长 η 起始 1e-3，按 dead-expert count 动态调整；前 2000 step 按 router_bias_norm、token_drop_rate、dead_expert_count 三条曲线监控 [10][11][17]。η 太大会让 bias 反复震荡，太小 dead expert 会锁死。
4. z-loss 不要和 aux loss 一起去掉。z-loss 管的是 router logit norm 的 fp16 overflow，与 aux loss 的负载问题正交 [5][11]。常用 1e-3。
5. dense→MoE upcycling 先问 dense 是不是 token-rich。只有 dense checkpoint 已超过 Chinchilla-optimal，upcycle 的有效 token 才能落在 0.4–0.6× [14]；token-poor dense 直接 upcycle 常常亏本，直接 from scratch 更稳。
6. Mixtral 阶段的 routing failure 要先 audit 再上线：context-independent routing、position-sensitive drop、drop-towards-end 三种模式在 OpenMoE trace 里都被放大 [12]。新 MoE 栈的 routing trace 要公开采样后回看一遍。
7. expert-choice 不要在 decoder-only autoregressive 场景落地——causality 被破坏 [4]；在 encoder-only / prefill-heavy 管线里它仍是值得保留的方案。
8. ≤13B active 的 serving 预算先走 dense。Olmo 2 / OLMo 3 [26][27] 给出的 dense ROI 在这个区间通常比同 active MoE 高，不要因为 MoE 时髦而强行上。

悬而未决的问题

  • Q1.MoE 在 SFT/RLHF 下相比 matched-quality dense 的稳定性差异尚缺公开可复现的 ablation——这是 Camp D 观点的关键证据缺口。
  • Q2.2024+ frontier scale 下 dense-vs-MoE 的 matched-quality ROI 对照实验几乎没有开源——OLMo 3 与 DeepSeek-V3 的不同训练 recipe 让直接比较含有过多混淆因子 [27][11]。
  • Q3.bias EMA 步长 η 的理论最优调度（1/√t 还是按 dead-expert count 自适应）仍开放——Han & Zhong [17] 给了收敛性证明但边界条件紧，工业日志之外缺少对比 benchmark。
  • Q4.shared expert 数量的甜点：1 vs 2 vs 4 在不同 active 预算下的 sweep 还不充分；DeepSeekMoE 默认 1 的选择需要 2025+ 更多实证支撑 [8][15]。
  • Q5.fine-grained MoE 的 serving 端：256 expert 的 memory/带宽瓶颈在消费级硬件上仍未解；ExpertFlow / MoETuner / MegaBlocks 只是 partial answer [21][22][6]。
  • Q6.task-specific expert pruning 与 dense-to-dynamic-k 转换能否扩展到 frontier scale？现有工作仍在 ≤30B total 的 scale 段 [23][24][25]。
