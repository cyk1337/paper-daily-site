上下文长度扩展的全栈工程：从”放得下”到”用得上”

Context Length Scaling in LLM Pretraining·24 篇论文·4 个子主题·2026年4月19日

作者@Thor·rescued

领域综述

长上下文不再是推理期的补丁。从 Llama 3 的 128K [37] 到 LongRoPE 的 2M [3]、再到 Nemotron-UL 的 4M 配方 [34]，位置编码、注意力算子、系统并行、数据工程和评估基准已被迫一起重新设计。本文把这条工程链拆成四个互相约束的轴：位置编码与外推（RoPE 插值系族 vs ALiBi vs NoPE）、注意力算子与架构（FlashAttention-2 [6]、MLA [7]、attention sink [9]）、系统并行（Ring [12]、Ulysses [13]、USP [14]、ByteScale [19]）以及训练/数据配方（渐进长度 warmup [31]、best-fit packing [33]、长依赖合成 [32]）。我们反对把长上下文工程当作单层问题：只做 positional interpolation 而不管 packing，上下文越长 loss 越虚；只堆 SP 而不调 RoPE base，RULER 会在 64K 之后塌陷。真正的判断是——架构、训练、系统、评估必须共同设计，评估基准 [LongBenchV2, RULER2024] 决定了你在哪一段是”真会用”。

位置编码、注意力算子、系统并行、数据配方——谁也不能单独跑赢长上下文

一句话导读

把”长上下文预训练”当成一条贯穿位置编码、注意力算子、系统并行、数据与评估的协同工程——任何一层的偷懒都会被 RULER/LongBench v2 [RULER2024, LongBenchV2] 放大。位置编码端，RoPE 插值系（PI [1] / YaRN [2] / LongRoPE [3]）在生产系统里全面压过 ALiBi，但 Kazemnejad et al. [4] 指出 NoPE 在外推上更稳——意味着”必须要有显式位置”的默认假设要打折。注意力算子侧，FlashAttention-2 [6] 把单卡 MFU 拉到 ~72%，Ring/Ulysses/USP [RingAttn, Ulysses, USP] 再把它扩到万卡，但 BurstEngine [20] 证明 >1M token 时现有 SP 会掉到 <30% MFU；ByteScale [19] 通过打破 SP/DP 正交假设恢复吞吐。KV 侧，MLA [DeepSeekV2, DeepSeekV3] 把 KV 缓存压到约 1/10 仍不退化，attention sink [9] 加上机制解释 [10] 则让流式推理在 4M token 上不崩。训练配方上，”长文档比例 + best-fit packing + 长依赖 SFT” [Fu2024DE128K, Ding2024Packing, Gao2024EffectiveLongCtx, Xu2025UltraLong] 比”堆 token”更决定性。一条更稳的读法：能放下不等于能用上，最终 bottleneck 从 GPU 显存转移到了评估基准。

核心断言

#1生产级长上下文的外推默认方案是 YaRN 风格非均匀 RoPE 插值 [YaRN2023, LongRoPE2024]；ALiBi 在 32K+ 的下游任务上已被实际工程放弃。
#2128K 下游能力由长文档比例决定而非 token 总量——≥25% 长文档 + 5B continual-pretrain tokens 就能饱和 NIAH [31]。
#3Ring/Ulysses/USP 在 ≤256K 稳居 SOTA，但 >1M token 后 MFU 跌到 <30%，必须上 BurstEngine / ByteScale 式动态负载均衡 [BurstEngine2025, ByteScale2025]。
#4MLA [DeepSeekV2, DeepSeekV3] 把 KV cache 压到 ~1/10 且不降质,是目前优于 GQA 的更优 KV 压缩方案。
#5perplexity 与 RULER/LongBench v2 的排名相关性弱 [Gao2024EffectiveLongCtx, LongBenchV2]，用 ppl 选长上下文模型已不再可信。

§1 位置编码:从 RoPE 插值到 ”要不要位置” 的再争论

RoPE [1] 以复指数形式把相对位置塞进 Q/K 的旋转里,短上下文收益明确;但训练长度之外,高频分量 θi−1​ 使注意力分数震荡。Position Interpolation [1] 选择把位置索引线性压缩回训练范围,用 1000 步短微调把 LLaMA 扩到 32K——思路朴素但代价是位置分辨率被整体稀释。YaRN [2] 的关键修正是”NTK-by-parts”:对低频维度做插值、对高频维度接近不动,再加 attention 温度校正,训练 token 需求降到 PI 的约 1/10,这也是 Llama-3 [37] 与 Qwen2.5 [39] 实际采用的配方。LongRoPE [3] 把 ”哪些维度插、插多少” 进化搜索化,一口气推到 2048K 且短上下文 perplexity 几乎不退化——它的经验结论是 RoPE 高低频的外推贡献严重不对称,统一缩放是浪费。XPos [5] 给 RoPE 加指数衰减以保持 attention resolution,定义了一个更诚实的外推诊断,但它训练成本高,生产侧没跟。

真正值得注意的反向证据来自 Kazemnejad et al. [4]:在小型 decoder-only 模型的合成外推任务上,NoPE(完全不加显式位置)反而稳过 ALiBi 和 RoPE。这条结论被常被忽略——但它指向一个更锋利的问题:causal mask 本身已经注入了足够的位置信息,显式 PE 可能是在训练分布里 ”拟合了位置” 而不是 ”泛化了位置”。工程侧的务实读法是:7B+ 生产模型继续用 YaRN 路线 [2],但做消融时,不要把 NoPE 当成学术玩具——如果你的任务是外推型而不是密集型,它值得作为 baseline 出现。ALiBi 在 32K 以下仍然是个便宜的选项,但过了 64K 它的线性衰减会把远距离信号压到接近 0,这是 Llama-3 [37] 选择 RoPE+YaRN 而不是 ALiBi 的直接原因。

RoPE 的高低频分量对外推的贡献严重不对称——统一缩放是在浪费容量。

§2 注意力算子与 KV 架构:FlashAttention 之后还有什么没解决

FlashAttention-2 [6] 把单卡 attention 的 MFU 从 FA-1 的 ~35% 拉到 ~72%,让”二次复杂度 = 不可行”这个老论断在 8K–64K 区间失去说服力。这背后的工程思路很干脆:tile 住 SRAM、不 materialize QKT、在 warp 级别重排 causal mask。FlashInfer [11] 把这套算子推广到推理侧的各种 mask/ragged 场景。但注意:FlashAttention 并不改变算法复杂度,它把常数压掉了;当 L 走到 1M+,常数优化碰到显存墙。

KV cache 才是长上下文真正的持续账单。MQA/GQA 通过共享 KV 头把 cache 缩到 1/nh​ 但以表达力为代价。DeepSeek-V2 [7] 的 MLA 走了一条更锋利的路:把 K、V 联合投影到低秩 latent,按 head 解压;同等 KV 预算下 context 能从 32K 扩到 128K,而 GQA 在此规模显示出明显的下游退化。DeepSeek-V3 [8] 把 MLA 推到 671B/37B-active 的生产级,是目前反驳 ”MLA 只在小模型好看” 的最硬证据。

推理侧还有一个常被低估的发现:StreamingLLM [9] 观察到前几个 token 是”attention sink”——即使它们语义无关,attention 仍会汇聚;只要保留首 4 token + 局部窗口,就能在 4M token 流式推理上稳定。Sun et al. [10] 给出了对应的微观机制:LLM 里存在少量比其它大 10^5 倍的 ”massive activations”,位置恰恰稳定对应 sink。这条机制解释让 attention sink 从 ”trick” 变成 ”结构性事实”,也顺带揭示了为什么单纯驱逐早期 token 的 KV 淘汰策略会崩。

一个被 attention-efficient 簇忽视的空白:Mamba/SSM 与 linear attention 在 4K–64K 区间与 softmax attention 的真实对抗证据仍然稀疏,open question 必须保留。

§3 系统栈:SP 互相之间的差别比它们跟 DP 的差别大

想在 12K+ GPU 上训 2048K 上下文,仅靠 ZeRO [29] 与 tensor parallelism [30] 够不到。序列并行(SP) 沿 L 切,是结构性解法——但怎么切决定了通信形状。Ring Attention [12] 把 Q/K/V 切块沿设备环流转,all-to-all 与计算重叠,显存从 O(L²) 降到 O(L²/N),causal 场景下 Striped Attention [17] 进一步消除了负载不均。Ulysses [13] 则换一种思路:attention 阶段切 head、其它阶段切 seq,用 all-to-all 换通信形状,单机内通常更快,但 head 数就是硬上限。USP [14] 的贡献不是发明新机制,而是证明 Ulysses 与 Ring 可组合成 2D 网格,按 head 数与带宽选布局——这一步在跨节点长序列上吞吐提升 1.4–2×。LoongTrain [15] 给出 head-context 2D parallelism 的另一个实现,是 USP 的工程同类。DistFlashAttn [16] 则把 FA-2 的 IO-aware 思想搬到分布式,是 Ring 路线上的补丁。

1M token 以上 SP 开始崩:BurstEngine [20] 实测 Ring/Ulysses 在 >1M 时 MFU 跌到 <30%,原因是 all-to-all 的包粒度与计算粒度错配。它通过 BurstAttention 的通信计算重叠 + 选择性检查点把 MFU 拉回 ~45%。更根本的修正来自 ByteScale [19]:它拒绝 ”SP 与 DP 正交” 这个沿用多年的假设——在真实变长语料上,把两者视为同一切分维度做动态 rebalance,才能在 12K+ GPU 规模保持 ~55% MFU。

显存轴上,gradient checkpointing [27] 与 selective recomputation [28] 是基础层;MEMO [25] 用细粒度 tensor 管理逼近理论下限;Mini-Sequence Transformer [26] 把长序列切成小序列迭代处理中间内存。StreamBP [22] 是一条值得单独标记的反向修正——它沿序列维分解链式法则,做到精确反传且 logits+激活峰值降到 ~1/8,否定了 ”长序列反传一定要近似” 的默认。OOMB [23] 把激活显存推到 O(1),代价是 ~1.7× wall-clock;Arctic Long Sequence Training [24] 把这些技术封装成 HF 生态可直接用的栈——这件事对学界比对 hyperscaler 更关键。

一条可执行的清单:≤256K 首选 Ring/Ulysses/USP [RingAttn, Ulysses, USP];>1M 必须上 ByteScale/BurstEngine 风格动态 rebalance [ByteScale2025, BurstEngine2025];反传内存吃紧时先换 StreamBP [22],offload 放在最后。

§4 训练配方与评估:从”放得下”到”用得上”

有了 128K 的架构与系统,数据配方才决定模型是否”真的会用”。Fu et al. [31] 的核心实验结论:128K 下游能力被长文档比例主导——按领域均衡地把长文档上采样到 ≥25%,5B continual-pretrain tokens 就能把 NIAH 打到饱和;把 token 数加到 50B 而不改比例,几乎没有额外增益。这条结论直接把”长上下文=烧更多 token”的朴素假设证伪。Xiong et al. [35] 更早指出,从 4K 到 32K 用渐进长度 curriculum 比一次性扩上去便宜 2–3 倍训练。

packing 是另一个容易被忽视的坑:naive 把文档拼接并按 L 切,会在长文档上切出虚假的 ”跨文档依赖”。Ding et al. [33] 的 best-fit packing 把截断率降 ~10×,下游长文任务 +1–3 pp——在 RULER 上这个 +1–3 pp 通常就是及格与不及格的差距。LongLoRA [36] 给出低成本微调路径:shifted sparse attention 让 7B 模型在 A100-8 上扩到 100K 仅需 8 小时。

评估端的真正转折是 Gao et al. [32]:在多个 128K 模型上,perplexity 的排名与 RULER/NIAH 的排名相关性很弱——这意味着用 ppl 挑长上下文模型已经不可信。LongBench v2 [40] 把评估从 NIAH 抽取升级到 8K–2M 的跨任务推理,暴露出多数 128K 模型在需要跨段推理时接近随机——这再次把 ”lost-in-the-middle” 推回议程。Xu et al. [34] 的 128K→4M 配方是当前唯一系统给出 ”长上下文扩展不伤短上下文” 证据的工作:4M RULER 下 NIAH >95%,短上下文 MMLU 仅降 0.5 pp。

一个务实的选型顺序是:先把 packing 与长文档比例校准好,再用 YaRN 风格扩展位置,再做 SP 训练;最后用 RULER + LongBench v2 双跑评估,别用 ppl。

perplexity 与 RULER 的脱钩是 2024 年最被低估的发现之一:它意味着过去的长上下文选型标准是错的。

研究立场对比

阵营 A:显式位置编码必需,RoPE 插值是唯一工程路线

立场 — 长上下文能力必须用某种学到/参数化的位置编码支持;ALiBi 与 RoPE 家族覆盖全部实际可行的选择。

证据：[1][2][3][37][39]

反方 — Kazemnejad et al. [4] 在合成外推任务上 NoPE 稳过 ALiBi 与 RoPE,说明 causal mask 已提供足够位置信号。

判词 — 结论层面的建议:7B+ 生产模型继续 YaRN 路线,但不要把 NoPE 当学术玩具;外推密集型任务下,NoPE 值得作为 baseline 出现在消融里。

阵营 B:SP 与 DP 正交,可以分别优化

立场 — SP 沿 L 切,DP 沿 batch 切,二者解耦;把它们分别压到极致即可。

证据：[12][13][14][15]

反方 — ByteScale [19] 实测:真实变长语料里,正交假设导致严重负载不均,在 12K+ GPU 上把两者统一为同一切分维度后吞吐恢复到 ~55% MFU。

判词 — 一个更稳的判断:≤256K 仍可把 SP/DP 当正交来调,>1M 必须当成一个维度动态 rebalance。

阵营 C:KV 压缩 = GQA/MQA 就够了

立场 — 共享 KV 头的 GQA 已经是 KV cache 压缩的合理终点,进一步压缩会伤下游。

证据：[38][37]

反方 — DeepSeek-V2 [7] 在同 KV 预算下把 context 从 32K 扩到 128K,V3 [8] 把 MLA 推到 671B/37B-active 仍不退化——是 GQA 的硬反驳。

判词 — 结论层面:新一轮 7B+ 预训练里,MLA 作为默认 KV 方案比 GQA 更合理,除非你的部署栈对 latent 解压不友好。

阵营 D:perplexity 仍然是长上下文的有效指标

立场 — 在 sufficient long 上训练,ppl 下降自然带来长上下文能力。

证据：[35]

反方 — Gao et al. [32] 与 LongBench v2 [40] 共同显示,ppl 与 RULER/跨段推理任务排名脱钩;ppl 下降 5% 的模型在 64K+ 任务上可能表现不升反降。

判词 — 结论层面:用 RULER + LongBench v2 双指标作为主评估,ppl 只用于 sanity check,不用来挑模型。

实践要点

一条可执行的清单,按优先级排序:
1. Do 把 YaRN [2] 作为 128K 外推的第一选项,2M+ 时切到 LongRoPE 的非均匀搜索 [3];Don't 在 32K+ 场景选 ALiBi,它在远距离会把信号压到 ~0。
2. Do 做 packing 审计:用 best-fit packing [33] 把截断率降 ~10×,并按领域把长文档比例校到 ≥25% [31];Don't 把 ”token 数不够” 当成长上下文不收敛的理由,真实问题多数是比例。
3. Do ≤256K 选 USP [14] 作为 SP 默认布局(头数够时偏 Ulysses、跨节点偏 Ring);>1M 立刻切 ByteScale/BurstEngine 式动态 rebalance [ByteScale2025, BurstEngine2025];Don't 为了规模一直堆 Ring,它在 >1M 会 MFU 崩塌。
4. Do 新一轮 7B+ 预训练默认上 MLA [DeepSeekV2, DeepSeekV3];Don't 在 128K 场景继续用 GQA,除非你的 serving 栈完全不能改。
5. Do 评估阶段用 RULER + LongBench v2 双指标 [RULER2024, LongBenchV2];Don't 拿 ppl 做长上下文模型排序,Gao et al. [32] 已证明相关性弱。
6. Do 反传显存吃紧时先换 StreamBP [22] 再考虑 offload;Don't 一上来就 activation checkpointing + CPU offload,wall-clock 代价太大。
7. Do 把 NoPE [4] 在外推密集任务里作为 baseline 跑一遍——用 0 成本换一条重要证据;Don't 把它当成 ”只在小模型上成立” 就略过。
8. Open — 证据不足:Mamba/SSM 与 linear attention 在 32K–256K 预训练规模上的硬对比、以及 H₂O/PagedAttention 类 KV 淘汰的机制性评估,目前文献仍薄,不要在这两条上下强断言。

悬而未决的问题

  • Q1.Mamba/SSM 与 linear attention 在 32K–256K 预训练规模上与 softmax attention 的硬对抗证据仍稀疏——是 softmax 已经够好,还是评估基准偏向 retrieval 掩盖了 SSM 的优势?
  • Q2.H₂O/PagedAttention 风格的 KV 淘汰与 MLA/GQA 的 KV 压缩在 128K–1M 推理下的真实 Pareto 还没被系统比较。
  • Q3.LongBench、NIAH、RULER、InfiniteBench 四者之间能否合成一个 ”真正使用长上下文” 的复合指标,而不是被单一基准牵着走?
  • Q4.sliding-window / local+global sparse attention(Longformer / BigBird / Mistral SWA)在 128K+ 预训练下是否仍具成本优势,还是已被 FA-2 + Ring 蚕食?
  • Q5.长上下文模型是 ”真的用上” 还是 ”只是放得下”?lost-in-the-middle 在 MLA、non-uniform RoPE 下是否减弱,仍缺乏机制层解释。
