数据配比：ratio 是 optimization，不是 lore

数据配比 Data Mixture·14 篇论文·4 个子主题·2026年4月19日

作者@Thor·rescued

领域综述

pretrain data mixture 在 2022 年之前更像玄学——Pile 的 22 域权重、The Stack 的手工比例、C4 的单一来源，都由 heuristic 决定。到 2023–2025，DoReMi [1]、RegMix [2]、ODM [3] 把它正式变成一个 constrained optimization 问题：domain weight w ∈ Δ^K，objective 是 downstream loss 或 worst-case group loss。与此同时 Llama 3 [5]、DataComp-LM [8] 公开的工程实践表明，mixture 不是一次性的静态向量，而是一条 curriculum——前段 web-heavy 稳住 loss surface，后段 upsample code/math/reasoning 来拉 capability。这份 report 的任务是把 data mixture 当作一个严肃的工程问题拆开：dedup 与 effective count 的前置、domain 粒度、ratio search 方法（heuristic / DoReMi / RegMix / online）、curriculum staging、以及 mixture 与 scale 的交互。结论层面给出一份可执行的 default，并直接点名当前社区仍在争论的几条轴：mixture search 是否值得 DoReMi 这种二次开销、online 方法是否稳定、以及在 quality filter 极其强的 regime 下 ratio 是否已经 degenerate 成 secondary。

把 mixture 当 optimization 问题处理，而不是一张配方表

一句话导读

到 2026 年，data mixture 已经不再是 Pile 时代那种手工配方。正确的工作流是四步：(1) dedup 先做，所有 ratio 都在 effective count 上谈 [11][7]；(2) 选一个与团队算力匹配的 ratio search 方法——小团队 heuristic + 2–3 轮 ablation，中团队 RegMix [2] 或 Data Mixing Laws [14]，大团队 DoReMi [1] 或 DoGE [4]；(3) 把 mixture 写成 curriculum，web-heavy 占 70–90% compute，capability-heavy 在 annealing 阶段 upsample code/math 3–5×，这是 Llama 3 [5] 与 MiniCPM [6] 的共同做法；(4) 守住稀缺域的 repetition 上限（≤4 epoch 近似免费，>16 epoch 塌缩）[12]。mixture 与 quality filter 共享同一条改进预算 [8][9]：当 filter 极强时 ratio 的边际贡献会收窄，但不会归零。

核心断言

#1RegMix [2] 以 ~30× 更低 compute 逼近 DoReMi [1]；中小团队应当默认选 RegMix，只有 >100B token、>10 domain 的场景才值得 DoReMi 的 proxy overhead。
#2Llama 3 [5] 和 MiniCPM [6] 的公开 recipe 把 capability-heavy stage 压在最后 10–30% compute；这个 schedule 不是美学选择，而是 LR cosine 尾段 gradient norm 下降时 upsample 难域的回报最高。
#3DCLM [8] 与 FineWeb-Edu [9] 的 filter 增益与 mixture 优化处在同一数量级（MMLU 3–6 pp）；这意味着两者竞争同一条改进预算，而非独立叠加。
#4稀缺域的 upsample 上限由 repetition law [12] 给定：4 epoch 以内近似等价新数据，超过 16 epoch 贡献塌缩；这把 math/code 在 mixture 里的 ceiling 固定在一个 hard 数字上。
#5Online mixing [3] 在固定 compute 下可持平 offline DoReMi，但稳定性依赖 loss signal 的 normalization；在强 quality filter 上游的 regime 下 online 容易退化为噪声驱动。

§1 把 mixture 写成优化问题

mixture 的正确写法是：给定 K 个 domain，domain weight w ∈ Δ^K，sampling 概率 pk​(w) = wk​，目标是最小化某个 downstream proxy L_down(θ(w))，其中 θ(w) 是以 w 作为 sampling 分布训出的模型。工程上所有分歧都可以归到三个问题：L_down 是什么、如何在小规模上估 L_down、小规模的 w* 能否 transfer 到大规模。DoReMi [1] 选择了 worst-case group loss（Group DRO），理由是 downstream 任务常常 ”稀疏域反而重要”；它通过一个与目标模型同架构、小一号的 proxy 训练出 w*。DoGE [4] 对此给出了一个有原则的修正：不盯 worst-case，而估计 domain 对 validation 泛化的 marginal contribution，这在 math/code 这类”小但有效”的域上给出的权重更合理。RegMix [2] 走另一条路：放弃 DRO 的目标函数，直接把 (w, L_down) 当作 tabular 回归样本，用 512 个 1M 模型上的随机 mixture 训一个 LightGBM，然后 argmin 预测。Data Mixing Laws [14] 进一步给出参数化 closed-form（每域一条 exponential），允许把 w* 外推到更大 N 与 D。这四种路线共享一个假设——小规模 w* 近似大规模 w*，即 ratio × scale 大致 invariant。这条假设在 [2] 的 1M→1B 和 [14] 的 cross-scale 实验中都被验证，但在 >30B 或 >10 domain 的 regime 下仍未被充分测试，是一个仍需警惕的 load-bearing 假设。

ratio × scale 近似 invariant 是所有 mixture search 方法共享的 load-bearing 假设。

§2 方法对比：DoReMi / RegMix / Online / Heuristic

四类方法可以沿四条轴定位——objective、proxy compute、transfer evidence、适用 regime。DoReMi [1] 的 proxy 与目标模型同架构、约 1/10 参数，proxy 训练量约等于目标模型的 10–20%，overhead 明显但给出可解释的 w*。RegMix [2] 用 512 个 1M 模型总共 ~30× 便宜，缺点是域数超过 ~20 时样本复杂度爆炸。ODM [3] 的 online 路线彻底绕掉 proxy，用 bandit 逐步更新 w，但对 loss signal 的 scale 非常敏感——不同域的 loss 绝对值不可比，normalization 做错会让 online policy 崩到单域。Heuristic (Pile、Llama 2 等) 的优势是零 overhead、可调 15–30 个域，劣势是给不出 ”再 upsample 1pp 会变好还是变坏” 的量化答案。下表把四种方法在 2026 年 frontier 团队关心的几个实际维度上横拉。

§3 Curriculum：把 mixture 从向量写成轨迹

Llama 3 [5] 的 pretrain 明确分为两段：前 85% 左右 compute 跑 web-heavy 稳态 mixture；最后 10–15% compute 进入 mid-train / annealing stage，code 被 upsample 约 3–5×，math/reasoning 类似，同时 LR 以 cosine 收尾。MiniCPM [6] 在更小的 scale（1B 级）重复了同一模板：decay stage 与 mixture shift 是同步事件，下游 MMLU/GSM8K 在这一段有阶跃式上升。这里的机制问题是：为什么不把 capability-heavy mix 从头用到尾？三条相互咬合的理由。第一，web-heavy 早期稳定了 loss surface——高质量但窄的域（math/code）在随机初始化附近 gradient variance 大，容易让 early train 发散。第二，repetition law [12] 告诉我们 math/code 这类稀缺域的 total-token budget 只够撑 4 个 epoch 左右；把它们堆在前段会在 compute 尚未收敛时就先耗尽 fresh-token 预算。第三，annealing 尾段 gradient norm 下降，模型对 ”信息密度高的 token” 的 per-step 利用率最高——这是 MiniCPM 的 decay-stage ablation 给出的直接证据。curriculum 并非 free lunch：它把 mixture 优化从 1D（单个 w）升到 2D（w 随 compute 变化的轨迹），搜索空间增大，DoReMi / RegMix 类方法需要扩展到 stage-wise 才能 cover。这是当前仍 open 的工程方向。

§4 Quality filter vs ratio：谁吃掉了谁的边际收益

DCLM [8] 的 7B controlled study 指出一个令人不舒服的结果：一条强 fastText classifier + perplexity filter 带来的 MMLU 增益 ~3–6 pp，与精心优化 mixture 的增益同一数量级。FineWeb-Edu [9] 在 web 端复现了这一结论——edu-classifier 过滤后的 corpus 在 MMLU 上的提升与一次完整 DoReMi run 接近。Phi-1 [10] 把这条路线推到极端：只用 textbook-quality + 合成数据训练，ratio 几乎不谈也能在 HumanEval 上达到 50%+。把这些证据串起来有两种读法。读法 A（”ratio is secondary”）：quality filter 对 effective token distribution 的重塑远比 mixture weight 剧烈，以至于 mixture 退化为 secondary knob。读法 B（更稳）：filter 与 ratio 竞争同一条改进预算——在 unfiltered Common Crawl 上 mixture tuning 能拿到 5 pp，在 DCLM-Baseline 上可能只剩 1–2 pp，但 filter + ratio 的总预算不是 5+3=8，而是更接近 max(5, 3)+小项。这解释了为什么 Llama 3 [5] 这种已经跑过强 filter 的 recipe，仍然在 curriculum 上投精力而不是再多刷一轮 ratio search——边际收益已经从 ratio 转移到 curriculum schedule 上。

§5 dedup 与 effective count：ratio 讨论的前置条件

在 dedup 之前谈 domain ratio 基本无意义。D4 [11] 给出了两个硬数字：document-level dedup + SemDeDup 带来 ~20% 训练加速；更重要的是 Common Crawl 的 near-duplicate rate 使得一个标称 ”web 70%” 的 mixture 实际 effective web 占比可能只有 40–50%，math/code 的占比被相对稀释。Dolma [7] 公开的 pipeline 显示，3T raw token 过完 MinHash + fuzzy dedup 后 effective token 落到约 1.6T，mixture 权重必须在这个 post-dedup 分布上重新标定才有意义。repetition law [12] 从另一端加一层约束：对稀缺域（math、specialized code），≤4 epoch 近似等价于新数据，>16 epoch 贡献塌缩。把两条加起来就给出一条硬规则——domain k 在 mixture 中的 ceiling ≈ 4 × (effective_tokens_k / total_train_tokens × K_current_ratio)。这条 ceiling 经常被忽视：一个”upsample math 到 8%”的决定，如果 effective math token 只有 10B、总训练 token 是 1T，就意味着 16 个 epoch，超过 [12] 给出的 ROI 拐点。Chinchilla [13] 的 N:D ≈ 1:20 在这里扮演锚点：没有这个 D 参照，”upsample 3×”这类描述无法归因到 loss。

研究立场对比

阵营 A：formal mixture search (DoReMi / RegMix / Data Mixing Laws)

立场 — domain weight 是可以 search 的变量；付出 proxy / regression 开销换一个量化最优的 w*，transfer 到大 scale 基本成立。

证据：[1][2][4][14]

反方 — 在 >10 域、>30B 规模上 transfer 的证据依然有限；DoReMi 的 proxy overhead 在 domain 不断变化的实际 pipeline 里要反复付费。

判词 — 一条更稳的读法：中团队默认 RegMix，大团队在 pretrain 的 main phase 投 DoReMi，annealing stage 单独再跑一次 search。

阵营 B：heuristic + curriculum（Llama 3 / MiniCPM 路线）

立场 — mixture 不是一个向量而是一条轨迹；手工先验 + 2–3 次 ablation 在 curriculum 视角下已经足够，关键在 stage 切分和 annealing mix，而不是 w* 精度。

证据：[5][6][12]

反方 — heuristic 无法回答”再 upsample 1 pp 变好还是变坏”；一旦 domain 数超过 20 或团队缺乏长期经验，heuristic 误差会快速累积。

判词 — 结论层面的建议：把 curriculum 当 first-class citizen，在 main phase 接受 heuristic 2 pp 的 slack，把 search budget 集中在 annealing mix 上。

阵营 C：online adaptive mixing

立场 — 把 domain 当 bandit arm，用 per-step loss signal 边训边调，彻底绕开 proxy training。

证据：[3]

反方 — 不同域的 loss 绝对值不可比；normalization 做不好会让 policy 塌缩到单域。目前开源复现仅限 <10B scale。

判词 — 在 quality filter 极强的 frontier regime 下 online 信号稀薄；建议作为 ablation / side experiment 而非主 pipeline。

阵营 D：ratio 不重要，quality 压倒一切

立场 — 一旦 filter 足够强（DCLM-Baseline、FineWeb-Edu、textbook-级 synth），mixture weight 就是 secondary knob，优化 budget 应当全部花在 content quality 上。

证据：[8][9][10]

反方 — Llama 3 [5] 即使跑过强 filter 仍然花精力在 curriculum 上；filter 不能替代对 multilingual / domain-scarce 数据的显式 upsample。

判词 — 更务实的定位：filter 与 ratio 争同一条预算而不是互斥，边际收益随 filter 强度递减但不归零；保持 mixture tuning 作为二级工具。

实践要点

一条可执行的清单（2026 frontier default）：(1) dedup 先做：document-level + MinHash + SemDeDup，所有 ratio 在 effective count 上报数 [11][7]；(2) domain 粒度切到 15–30，web 按 quality tier、code 按语言、math 按难度、multilingual 按语言；(3) default 起点：web 60–75% / code 10–20% / math 2–8% / multilingual 5–15% / book+wiki 3–10% / synthesis 5–20%；(4) ratio search 按团队规模选——小团队 heuristic + 2–3 次 ablation，中团队 RegMix [2] 或 Data Mixing Laws [14]，大团队 DoReMi [1] 或 DoGE [4]；(5) 把 mixture 写成 curriculum：main phase 70–90% compute 跑稳态 web-heavy，annealing stage 10–30% compute upsample code/math 3–5×，与 LR cosine 尾段同步 [5][6]；(6) 稀缺域 upsample 前核对 repetition ceiling：effective_tokens × 4 ≥ planned_tokens_in_mix，否则按 [12] 会踩 ROI 拐点；(7) 不要把 mixture 优化和 quality filter 优化放到同一预算里独立叠加——它们竞争 [8][9]，先锁 filter 再 tune ratio；(8) 不要在 main phase 之外去跑 online adaptive mixing [3]，除非你有稳健的 per-domain loss normalization——否则它会退化为噪声驱动的域塌缩。

悬而未决的问题

  • Q1.ratio × scale invariance 在 >30B 参数、>10 域的 regime 下是否仍近似成立？现有 transfer 证据主要停在 1M→3B。
  • Q2.Curriculum 的 stage 数和切分时间是否也可 search？目前 Llama 3 / MiniCPM 的 ”85%-web + 15%-capability” 仍是经验数字。
  • Q3.在 DCLM-Baseline / FineWeb-Edu 这种强 filter 上游之后，mixture 优化的边际收益 ceiling 是多少？当前没有干净的量化答案。
  • Q4.Online mixing 在 quality filter 稀疏化 loss signal 的 regime 下能否稳定？ODM 的开源复现仍限于 <10B。
  • Q5.合成数据在 mixture 里该占多少？repetition law 的 epoch 概念对 synth 是否还适用，仍是空白。
