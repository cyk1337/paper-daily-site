16:T431,In 2026 three parts of the pretrain data pipeline have crystallised from ad-hoc choices into a publishable engineering blueprint: packing uses per-doc causal mask (native in FA2/FA3 varlen); length uses a short-to-long curriculum (85–95% of compute at short window, final 5–15% as mid-train at 128K+); FIM is reserved for code models. LLaMA-3 [Llama32024], Qwen2.5 [Qwen25Tech] and StarCoder [StarCoder2023] tech reports converge on these choices, yet many teams still run naive concat with cross-doc visible attention, or apply FIM as a default augmentation to NL models. This report treats packing / masking / length / FIM as one data pipeline: why cross-doc contamination became a zero-cost fix after FA2 varlen [FlashAttention2Varlen], how short-to-long compares in compute accounting against mixed-length, what RoPE base range to use in the long-context stage, and why FIM is not recommended for NL models. Target audience is researchers and infra engineers building or refactoring a pretrain data pipeline from scratch, with an executable 2026 default at the end.17:T55e,In 2021 Krell et al. [Packing2021] first turned 'cross-doc loss contamination' into a measurable question: when multiple docs are concatenated into one seq without masking, early tokens of a later doc attend to end tokens of the previous doc, artificially depressing the loss at early positions by about 0.5–2% (varying with doc length distribution). Most 2022–2023 open models, including GPT-2 [GPT2] and the original LLaMA [LLaMA2023], chose EOS-only separators with cross-doc visible attention — not out of ignorance of contamination, but because attention kernels did not support cheap in-sequence masking at the time, and a 20–30% throughput hit was harder to accept than a 1% loss bias. The FA2 varlen API [FlashAttention2Varlen] reversed the accounting: cu_seqlens hands per-doc boundaries to the kernel, inter-doc attention is masked at the kernel level, throughput loss is < 3% and packing ratio stays > 98%. FA3 [FlashAttention32024] extends this interface to Hopper and FP8, confirming it as a long-term engineering contract. LLaMA-2/3 [Llama32024] adding per-document mask is effectively a correction of its own earlier recipe, a tacit admission that the 2023 EOS-only version is obsolete. The conclusion is simple: any new pretrain pipeline in 2026 should default to per-doc causal mask; the only remaining debate is the boundary-token design (see §4).18:T5a6,There are only two real candidates for length curriculum: short-to-long (short-window main pretrain plus long-context tail stage) vs mixed-length (interleave short and long seqs across the whole run). LLaMA-3 [Llama32024] publishes ~95% of tokens at 8K with a final 128K long-context mid-train; Qwen2.5 [Qwen25Tech] uses a similar stage structure and raises RoPE base from a default 10K to 1M. The reason is compute: attention is O(L²), so spreading compute evenly over short and long seqs forces the 80% of docs shorter than 4K into a long-doc-sized expensive window, adding 20–40% wallclock vs short-to-long. YaRN [YaRN2023] supplies the matching RoPE frequency rescaling and attention-temperature correction so the tail stage needs only ~100B tokens to stabilise perplexity in the 32K–128K range. Two objections remain: that the tail mid-train is insufficient to teach 'how to use long context', and that attention-head distributions trained at short window diverge from the long-context optimum. Public reports suggest the first is mitigated by mid-train budgets ≥ 100B tokens plus synthetic long-context data, while the second, once RoPE base is large enough, does not appear on needle-in-haystack benchmarks (LLaMA-3 128K sits close to Qwen2.5 [Qwen25Tech] on RULER-style evals). A pragmatic recommendation: default short-to-long, 85–95% / 5–15% compute split, RoPE base 500K–1M in the tail, long-context token budget ≥ 100B.19:T48b,OpenAI FIM [FIM2022] claims that reordering a doc into prefix / middle / suffix and training infilling at 50–90% probability barely degrades left-to-right loss while greatly improving infilling — the so-called 'free lunch'. The catch is evaluation: the original NL evals are shallow, and subsequent independent reproductions on the code side (StarCoder [StarCoder2023], Code Llama [CodeLlama2023]) all confirm significant code gains, while Code Llama observed a small but consistent NL regression (1–2 pp on MMLU / commonsense). This means the 'free lunch' holds on code-heavy data but is not strictly free on NL-dominant pretrain. A second hidden cost is tokenizer / sentinel budget and the need for a consistent downstream infer interface, rarely worthwhile for an NL-only model. A pragmatic stance: default 50% FIM for code models with StarCoder span selection; no FIM by default for NL models, and if edit-like capabilities are needed, handle them in mid-train plus instruction stages. Does this contradict the original FIM paper? Not really — its evidence scope was code plus light NL; the generalisation to NL-only pretrain was always on the reader.1a:T4a2,Boundary design for packed seqs has three small but important decisions: (1) EOS-only vs EOS+BOS; (2) whether the first-token loss of each packed doc is counted; (3) how to handle docs exceeding max_seq_len. GPT-2's [GPT2] <|endoftext|> + EOS-only tradition was inherited by LLaMA [LLaMA2023], but the post-2024 default is BOS + EOS double markers, because once per-doc masking is on, the first token of each doc has no context to attend to and its loss mostly reflects unigram frequency, which is noisy. An explicit BOS gives the model a stable 'doc start' signal, and dropping the first-token loss from the objective makes the training signal cleaner — NeMo's [NeMoPacking2024] public implementation binds these two together. For over-length docs, split-then-pack (split at max_seq_len, pack each chunk independently) retains all tokens and produces more stable downstream long-context behaviour than truncate-and-drop; LLaMA-3 [Llama32024] and Qwen2.5 [Qwen25Tech] point this way in their public descriptions. The only open question is whether to keep a small overlap at split boundaries to soften edge effects. Public evidence is thin, so treat it as an experiment, not a default.1b:T4a3,An executable checklist (2026 pretrain data pipeline default):
1) Default packing with per-doc causal mask via FA2/FA3 varlen API [FlashAttention2Varlen][FlashAttention32024]; target packing ratio > 98%.
2) BOS before and EOS after every doc; reserve both special tokens and do not reuse a single <|endoftext|> (a GPT-2 [GPT2] legacy).
3) Drop the first-token loss of each packed doc from the objective; follow NeMo's [NeMoPacking2024] implementation.
4) For docs exceeding max_seq_len, split-then-pack, never truncate-and-drop [Llama32024][Qwen25Tech].
5) Length curriculum defaults to short-to-long: 8K short window for 85–95% of compute, tail 5–15% for 128K long-context mid-train [Llama32024].
6) In the long-context stage move RoPE base from 10K to 500K–1M and apply YaRN's [YaRN2023] attention-temperature correction; budget ≥ 100B mid-train tokens.
7) FIM only for code models, 50% rate with StarCoder [StarCoder2023] span selection; NL models default off, with Code Llama's [CodeLlama2023] 1–2 pp NL regression as primary evidence.
8) Mixed-length training is not the default; adopt it locally only when long docs genuinely dominate (legal, large monolithic
