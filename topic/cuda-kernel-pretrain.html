CUDA Kernel × Pretrain 算法：共同演化、诊断与 2026 前沿方向

CUDA Kernel × Pretrain Algorithm co-evolution·22 篇论文·4 个子主题·2026年4月19日

作者@Thor·claude-opus-4.7

领域综述

近三年 pretrain 架构的选择越来越像 kernel 物理的投影：GQA / MLA 是 decode memory-bound 的直接产物；fine-grained MoE 生于 grouped GEMM；FP8 训练绑死在 MXFP8 per-block scaling 上；RoPE head_dim 实际上被 tensor core tile 约束在 {64, 128, 192, 256}；sliding-window attention 依赖 FlexAttention / FA3 才能跑出规整 MFU。这不是偶然，而是 H100/B200 的 SM、SMEM、HBM、TMA、async tensor core 几条硬件约束沿着 roofline 反推出的结果。本报告的目标不是再一次罗列 kernel 技巧，而是把 2022–2026 之间 kernel 与 pretrain 算法的共同演化轨迹梳理成可诊断、可操作的框架：哪些架构变化是 kernel 逼出来的、哪些是算法自主选择、哪些看起来是算法创新其实只是在适配新一代 tensor core；诊断任何 kernel 问题的正确起手式是什么（答案：先 roofline，再 Nsight，再 fusion / tile / CUDA Graph）；2026 的 frontier pretrain team 最低需要哪些能力配置。

从 SM/SMEM/HBM 到 MLA/MoE/FP8：kernel 物理如何塑造 pretrain 算法

一句话导读

2022–2026 年 pretrain 架构的主流变化 70% 以上可以由 kernel 约束直接推出：GQA/MLA 来自 decode memory-bound（[4]、[5]、[6]）；FlashAttention 系列是 SMEM tiling + Hopper async 的工程化（[1]、[3]）；FP8 训练只有在 MXFP8 per-block scaling 上才稳定（[10]、[11]），否则到 trillion-token 就会漂移（[12]）；fine-grained MoE 能落地是因为 grouped GEMM 成熟（[22]）。但这不意味着 PyTorch 层面的算法师可以不管 kernel——[7] 已经显示 FlashAttention 的 numerics 偏差会外泄到 loss spike。一条更稳的读法是：2026 的 frontier pretrain 团队最低配置是算法 + kernel + profiling 三者同时在编；诊断任何 kernel 问题的第一步永远是 roofline ([14]) 定位 memory/compute/latency bound，而不是盲目加 fusion。

核心断言

#12022–2026 pretrain 架构 70%+ 的关键决策由 kernel 约束驱动，而非独立算法选择：GQA/MLA 的 KV head 数 ([5]、[6]) 直接来自 decode memory-bound 的 bandwidth wall。
#2FP8 pretrain 在 100B+ 规模下没上 per-block scaling 会漂移——[12] 在 2T tokens 暴露了 per-tensor scaling 的系统性失败，[11] 给出可复现的 MXFP8 解。
#3Kernel 的 numerics 选择会外泄到 convergence：[7] 显示 FlashAttention 的分块 softmax 与大规模训练的 loss spike 存在相关性，打破了'kernel 只影响速度不影响质量'的错觉。
#4诊断任何 kernel 的正确起手式是 roofline ([14])，而不是 fusion：memory-bound 的 kernel 做算子融合几乎没用，必须先分类再选优化。
#5Fine-grained MoE（DeepSeek-V2/V3、Mixtral、Qwen-MoE）是 grouped GEMM 成熟 ([22]) + Hopper async ([19]) 的结果，而不是先有算法灵感再找 kernel。

§1 Attention kernel 与架构的共同演化

把 attention 的历史按 kernel 约束重画，会发现算法创新几乎是按 SMEM → async → KV compression 的顺序被硬件逼出来的。[1] 把朴素 attention 从 HBM-bound 搬到 SMEM-tiled IO-aware，前提是 Ampere 的 SMEM 够大、softmax 可以分块稳定重组；Dao et al. [2] 把 warp-level 的 work partitioning 改进到再 ~2×；Shah et al. [3] 在 Hopper 上把 async tensor core + TMA + warp-specialization 三件套吃透，FP16 逼近 75% peak。这条线不是算法家坐下来想出的，而是 [19] 给出的 microarch 证据回推出来的。与此同时，decode 侧 memory-bound 几乎永恒：per-token FLOPs 小，per-token KV load 大，所以 Shazeer [4] 先把 KV 头砍到 1，Ainslie et al. [5] 把它插值回 8/4，DeepSeek [6] 的 MLA 再用低秩 latent 压到等效 ~9 KB/token。三者是同一条 bandwidth wall 上的三次减法，而不是三条独立思路。值得警惕的是 Golden et al. [7]：FlashAttention 的分块 softmax 累积误差与 trillion-token 规模下的 loss spike 相关——kernel 的 numerics 从来不是纯工程问题。

MQA → GQA → MLA 是同一条 bandwidth wall 上的三次减法，而不是三条独立思路。

§2 低精度训练：从 loss scale 到 MXFP8 per-block scaling

Mixed precision ([8]) 确立了 'FP32 master weight + FP16 compute + loss scale' 三件套，此后每一次精度下探都是在挑战同一个稳定性模板。FP8 ([9]) 定义 E4M3/E5M2 两档格式和 per-tensor / per-channel / per-block 三级 scaling 时就已经暗示：per-tensor 在大模型上不够用。Fishman et al. [12] 把 FP8 推到 2T tokens 时正面撞到这堵墙——Hopper 只支持 per-tensor scaling，SwiGLU 的 activation outlier 在 trillion-token 累积成 loss spike。他们打的补丁是 'Smooth-SwiGLU' 与 Adam 动量 FP8 化。真正的结构性解由 OCP MX 联盟 [10] 提出：32-element block 自带 8-bit 共享指数，Blackwell 的 5th-gen tensor core 直接硬件支持。Mishra et al. [11] 给出 NVIDIA 官方 MXFP8 pretrain recipe，明确哪些 tensor 下 MXFP8、哪些保留 FP32 master、scaling 如何与 optimizer update 交互——这是目前 Blackwell FP8 训练的事实标准。Xi et al. [13] 顺势把 FP8 扩张到 optimizer state 与 activation，让 memory 真正接近理论 1/2。我们给的操作边界：未上 MXFP8 的 FP8 pretrain 过 ~100B 规模通常会崩，H100 上即使做 per-channel scaling 也只是延迟崩溃；FP4 forward + FP8 backward 的 hybrid 是 2026 下半年 frontier 的默认实验，但量产还早。

§3 诊断学：roofline 先行，fusion 在后

一个常见的错误是遇到慢的 kernel 就往 fusion 上扑。Williams et al. [14] 18 年前给出的方法论今天仍然是对的：先算 arithmetic intensity = FLOPs / bytes，放到 peak-FLOPs × bandwidth 的 roofline 上分类。结果分三类：memory-bound（GEMV、decode attention、RMSNorm）、compute-bound（长 prefill GEMM、FA3 forward）、latency-bound（小 kernel 接力、Python overhead、NCCL launch）。诊断配方也明确：memory-bound 上 fusion / register reuse / bf16→fp8 压 bytes（但要防 [7] 里的 numerics 泄漏）；compute-bound 调 tile size、warp specialization、async copy（参考 [16]、[3]）；latency-bound 最好用 CUDA Graph 把 launch 合批，或用 Triton ([15]) 减 kernel 数。Pope et al. [25] 把 decode / prefill 的 memory↔compute 切换点量到 batch × seq 的函数，是 serving 侧的工程教材。一个实操原则：新架构或新 layer 上线前必须跑一次 Nsight Compute + roofline，MFU 变化 > 2% 要 diff profile 并写结论——否则性能回归会沉底。中小团队不必啃 CUTLASS，FlexAttention + torch.compile 已能覆盖 80% 场景；frontier 团队则需要至少 1 位能读 CuTe-DSL 的 engineer。

§4 并行 × MoE：grouped GEMM 与 cluster 带来的算法红利

TP/PP/SP 的设计本质是带宽预算的分配问题。Shoeybi et al. [21] 给出的 row/column TP 至今仍是 baseline；Korthikanti et al. [20] 把 SP 引入 LayerNorm/Dropout 切分，并系统分析 TP × PP × SP 下的通信重叠；[32] 530B 是这套 3D 并行的极限案例。从 Hopper 开始，async tensor core + TMA + thread-block cluster ([19]) 让 SM-到-SM 的 DSMEM 传输变得可编程，直接降低 grouped GEMM 的启动开销——这是 fine-grained MoE 落地的 kernel 前置条件。Gale et al. [22] 的 block-sparse grouped GEMM 把 dropless MoE 做成训练原语，之后 DeepSeek-V2/V3、Mixtral、Qwen-MoE 的 256-expert 级 fine-grained MoE 才成为可能。长上下文训练侧，Brandon et al. [23] 把 Ring Attention 在 causal mask 下的 load 不均用 stripe permutation 抹掉，per-step 时间降 ~1.45×；Liu et al. [24] 进一步把这套范式推到百万 token pretrain。这些工作的共同点是：算法更新的边界由 kernel 启用的新原语划出，而不是反过来。

研究立场对比

阵营 A：kernel 与算法必须 co-design

立场 — 架构选择是 kernel 物理的投影：GQA/MLA、FP8-per-block、grouped GEMM、head_dim ∈ {64,128,192,256} 都由 roofline / tensor core tile / memory hierarchy 推出。

证据：[1][3][5][6][22][11][19]

反方 — 对非 frontier 规模团队成本过高；小团队不需要自写 kernel 也能拿 80% 性能。

判词 — 一条更务实的定位：frontier 团队必须 co-design；200B 以下工程队靠 FlexAttention + torch.compile + 一位 Nsight 熟手即可。

阵营 B：PyTorch 层就够了

立场 — torch.compile + FlexAttention + 标准 transformer 原语已经吃下大多数 kernel 增量；算法师不需要读 CUTLASS。

证据：[15][34]

反方 — [12] 显示 trillion-token 规模必须动 kernel-level numerics；[7] 显示 kernel 选择会泄漏到 convergence；MLA ([6]) 需要自定义 attention 实现。

判词 — 结论层面的建议：这个阵营在 ≤ 70B 的 post-training / 微调场景基本成立；在 pretrain frontier 已经失效。

阵营 C：硬件会继续进步，算法不用适配

立场 — Blackwell / Rubin 每代 2× bandwidth + 2× FLOPs，scale-up 足以让 dense MHA + BF16 继续跑，算法复杂化是自找麻烦。

证据：[18]

反方 — 每代硬件的 capability 新增反而强制算法迁移：Hopper async 催生 FA3，Blackwell MXFP8 催生 [11] 配方，FP4 tensor core 推动 hybrid 精度。算法不是不用变，而是被新 primitive 逼着变。

判词 — 一条更稳的读法：硬件进步不是 'algorithm 闲下来'，而是 'algorithm 被迫重写'；这个阵营在可观察数据上基本证伪。

阵营 D：非 NVIDIA 硬件会追上

立场 — TPU v5p / MI300X / Trainium2 会在 2026 内把 frontier pretrain 的硬件依赖多元化，kernel ecosystem 也会随之分散。

证据：[33]

反方 — CUTLASS 3 / CuTe-DSL ([16]) 与 FA3 ([3]) 的 software moat 仍在扩大；MXFP8 硬件支持目前仅 Blackwell，非 NV 家还在追 FP8 per-block。实测 frontier ecosystem 仍明显 NV 为主。

判词 — 一条务实的定位：2026 内 NV 份额仍压倒性；非 NV 对 pretrain 的主要价值是 throughput / 成本 backup，而非 kernel frontier。

实践要点

一份可执行的 2026 清单：

1. 团队配置硬底：frontier pretrain team ≥ 1 位能读 CUTLASS 3 / CuTe-DSL 的 engineer + ≥ 1 位 Nsight Compute 熟手；中小团队 FlexAttention + torch.compile + Triton ([15]) 覆盖 80%。
2. 诊断 = roofline 先行：任何 kernel 先算 arithmetic intensity，在 [14] 分类为 memory/compute/latency bound 再配优化；不要直接上 fusion。
3. 新 layer 必须 profile：架构或 layer 落地前跑 Nsight + roofline；MFU 变化 > 2% 要 diff profile 并写结论。
4. FP8 默认 MXFP8：Blackwell 生产默认 MXFP8 per-block ([11])；H100 上做 FP8 pretrain 上限 ~100B，[12] 的 Smooth-SwiGLU 补丁是必要条件。
5. FP4 的节奏：2026 Q2 起在 < 10B 规模做 FP4 forward + FP8 backward hybrid 实验；量产至少 2027。
6. head_dim 严格 ∈ {64,128,192,256}：非标准 dim 必须 profile + 边界 padding；不要让 tensor core tile miss 侵蚀 MFU。
7. KV 压缩优先级：decode 场景从 GQA → MLA → 2-bit KV ([26]) 梯度选用；RWKV 类线性递归 ([28]) 目前只做 hybrid 不做主干。
8. 不要把 kernel 改动当纯工程：[7] 已表明分块 softmax 会泄漏到 convergence；任何 kernel 改写上线前要跑 ≥ 10B token 的稳定性回归。

悬而未决的问题

  • Q1.PyTorch 2 torch.compile + FlexAttention 在 100B+ pretrain 规模的极限在哪？目前公开文献几乎空白（证据 sparse）。
  • Q2.'硬件进步让算法不用变'这个立场是否在 Rubin 这一代真正站得住？需要 dense + BF16 baseline 在 Blackwell / Rubin 上跑完整 scaling curve 的公开数据。
  • Q3.TPU v5p、MI300X、Trainium2 的 frontier-scale pretrain report 稀缺；非 NV kernel ecosystem 是否会分叉仍未定。
  • Q4.FlashAttention numerics 的长期影响：哪些 loss-spike 真正由 kernel 偏差导致、哪些是 data / optimizer 问题？需要 controlled ablation。
  • Q5.FP4 forward + FP8 backward 的最小可行 recipe 尚未公开收敛，MXFP4 的 block scaling 粒度也仍在实验。
  • Q6.Sliding-window attention（Mistral、Longformer 系）在 FlexAttention / FA3 下的准确 MFU 曲线公开数据稀缺，长上下文 pretrain 的 kernel 边界未被严谨标注。
  • Q7.线性递归 ([28]、[29]) 与 Transformer hybrid 的最优比例在 100B+ 规模仍无定论。
